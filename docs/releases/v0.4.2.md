# v0.4.2 — Governance & Integration

**Theme:** close the dual-control loop — make it structurally impossible for a human approval alone to bypass the overseer gate, and make the overseer the one who executes the merge once both gates are satisfied.

This release delivers the three named governance charter items (protected surface, required status check, overseer-executes-merge), plus a critical consumer-install fix, a cluster of security/fail-closed hardening, four cron-launcher performance wins, and a long tail of portability and process-gap closures.

---

## Headline: dual-control loop is closed

The `bin/` cron launchers are now **a declared protected surface**: any change to them must pass the overseer gate before merging, with human authorization required. The require-overseer-approval GitHub status check is wired in, so a human approval alone cannot bypass the gate — the check blocks until the overseer signs off. And when the overseer finds an approved, gate-passing protected PR, it now **executes the merge itself**: the human authorizes, the overseer acts — neither can complete the merge unilaterally.

- **`bin/` as protected surface** — declared in `AGENTS.md` and wired into branch-protection rules; cron launchers and identity-guard code are control-defining and require dual-control (#744).
- **`require-overseer-approval` status check** — GitHub-reported required check blocks merge until overseer posts an approved verdict; human approval alone is no longer sufficient (#621).
- **Overseer executes the merge on protected PRs** — once a verified human approval exists and the gate clears, the overseer merges; authorization and execution are separated (#589, #741).
- **`needs-human` label guard** — overseer merge logic now checks for `needs-human` before auto-merging; previously the label was not consulted and gate-passing PRs with the label could auto-merge (#756, #757).

## Consumer install: critical completeness fix

- **Installer ship-set completed** — `hos_install.sh` was missing `bin/lib/`, `.github/workflows/`, and `scripts/framework/` tools; consumers got a broken cron launcher and non-functional branch-protection setup. All paths now installed. This was a silent ship defect in v0.4.1 (#769).

## Security & fail-closed

- **`run_second_review.sh` no longer sources `.env` as shell** — the script executed a repo-controlled `.env` file as `bash`, enabling an arbitrary code-execution vector; now reads environment variables directly without source (#765).
- **`setup_clis.sh` + `django_check.sh` fail closed on missing tool** — both scripts previously reported success even when a CLI was absent or a setup step failed silently; now fail with an error and do not report a false pass (#764).
- **`run_second_review.sh` reviewer selection no longer fails open** — `SELECT_OUT` eval error left the reviewer slot empty; the slot is now checked and the review errors out rather than running with a missing reviewer (#681).
- **`run_panel.sh` required reviewers can no longer skip** — a reviewer that exits early or is absent now blocks the panel from converging clean (#682).
- **Overseer artifact verification hardened** — full implementation of the artifact-read gate; overseer now reads and validates the committed validator artifact before making any merge decision, not just checking CI status (#695).

## Cron-launcher performance

Four optimizations that reduce the cost of idle cycles and speed up the common paths:

- **Skip Claude launch when all open worker PRs are awaiting human merge** — if the only open PRs are approved/needs-human, there is nothing for the worker to do; the cron launcher exits before starting a Claude session (#791).
- **Cycle context pre-computed in `hos-cron`** and injected directly into the Claude prompt — open PRs and next-work candidates are fetched before Claude starts; no API calls needed inside the turn for that data (#792).
- **Halt check before Claude launch** — `hos-cron` checks for open `hos-halt` issues before starting Claude; if a halt issue is open, the launcher exits immediately (#793).
- **Agent file availability check before Claude launch** — launcher verifies that the required cron-prompt files exist before launching Claude; avoids a session that would fail immediately (#794).

## New capabilities

- **`hos-suspend` CLI** — `bin/hos-suspend <project>` pauses a project's autonomous work without killing cron; leaves cron running and the loop intact but suppresses worker cycles until unsuspended (#778).
- **GitHub Actions slash-commands** — `/approve` and `/handoff` label-swap commands in GitHub Actions for the human reviewer workflow; reduces friction for the common approval and deferred-to-human handoff paths (#601).
- **Target milestone de-hardcoded** — the worker's target milestone/release is now read from `projects.conf` and injected into cron prompts at launch; rolling to the next milestone is a one-line change in config, no file edits needed (#780).
- **Probe milestone-model fix** — `probe.py` was querying via `hos-coordination` label, incompatible with the milestone-based model; now uses the milestone API correctly (#619).

## Portability & correctness

- **Stale FAILURE check runs no longer block merge** — when newer successful runs exist for a required status check, old FAILURE runs are filtered out; previously a stale failed run would block an otherwise-passing PR indefinitely (#795).
- **Cron-prompt files de-hardcoded** — machine-specific path literals in the injected prompt files replaced with `REPO_ROOT`-relative paths set at launch time (#805).
- **`hos_bootstrap.sh` now writes the bootstrap marker** — the marker that `hos-cron` checks was never written; cron logged a warning every cycle. Fixed (#806).
- **`ensure_venv.sh` marker checks Python version and venv path** — the trusted-forever marker would hide a broken venv after a Python upgrade or path change; now validated on each use (#807).
- **Scripts portability cluster** — `run_panel.sh` path fix, installer `set -u` array portability, `mktemp` portability, self-review path fixes (#766).
- **Bash enforcement** — all framework scripts now have `#!/usr/bin/env bash` shebangs and pass shellcheck under Bash 3.2 rules; Bash-ism portability guardrail added to CI (#768).
- **Worker env-validation fails closed** — worker validates its required environment variables before the jitter sleep; missing env now blocks before any work starts rather than failing mid-cycle (#774).

## Cleanup

- **Legacy orchestrator retired** — `hos_worker.sh` and `hos_orchestrator.sh` (the pre-cron orchestration path) removed; all active code paths use `bin/hos-cron` (#763).
- **`.envrc` fixed** — `direnv_layout_dir` call replaced with `BASH_SOURCE[0]`-relative path; `.envrc` now works on fresh clones without a prior `direnv allow` (#773, #770).

## Process gaps closed

- **Merge-audit detects bot self-merges** — Step 0 merge-audit was reading `merged_by` from the list endpoint (always null); now fetches per-PR detail endpoint where `merged_by` is populated; bot self-merges are now detectable (#758).
- **Needs-human label checked before auto-merge** — gap that allowed the overseer to auto-merge PRs labeled `needs-human` without human authorization (#756, #757).
- **Escalation comments no longer post literal `@path`** — a formatting bug caused escalation comments to post the raw `@variable` path string instead of the content; now posts the content (#752).
- **PR attribution format reconciled** — `AGENTS.md` and worker practice aligned: the `[AI: <login>]` prefix format is now consistent in both docs and in every PR the worker opens (#754).
- **NG3b incident closed** — verified that #744+#621 structurally prevent the class of incident where the overseer auto-merged a human-gated release artifact (#761).
- **Unintentional bot self-merge pattern documented** — the #773 merge audit (PRs #775–#777, #781) triggered a governance review; the gap was closed by the #756+#757 label-guard work above.

## Known limitations

- **Duplicate audit PRs** (#849) — when a worker PR is blocked across multiple cycles, the overseer may create a new audit PR per cycle rather than checking for an existing one. Produces audit PR spam. Deferred to v0.4.3 / v0.5.0.
- **Worker branch stacking** (#850) — worker occasionally builds a feature branch stacked on an in-progress fix, resulting in a PR that includes commits from both. Deferred to v0.4.3 / v0.5.0.

## Upgrade notes

- **Branch protection**: re-run `./scripts/framework/setup_branch_protection.sh <owner/repo>` to register the `require-overseer-approval` status check as a required context. Without this, the new check gates nothing.
- **Consumer reinstall required**: re-run `./bootstrap/hos_install.sh` to pick up the previously missing `bin/lib/`, `.github/workflows/`, and `scripts/framework/` paths (#769).
- **`projects.conf`**: ensure `hos_target_release=v0.4.2` (or the next target) is set; the worker no longer reads the milestone from hardcoded prompt files.
- **`.envrc`**: if using direnv, `direnv allow` after the `.envrc` fix (#773) — the layout function changed.
