# v0.3.5 — manifest fail-close + version-as-commit-point, and runbook accuracy

Completes the "installer never reports success over a swallowed failure" theme (v0.3.2), plus two runbook-accuracy fixes.

## #277 — manifest assembly fails closed; the version stamp is the commit point

Two post-Phase-B manifest-assembly calls suppressed stderr / swallowed failures. Because they run *after* the agent files are written, a crash there couldn't corrupt the agents — but it could write a degraded `.hos-manifest` (missing every agent's REGION rows) and still stamp the new `.hos-release`, silently breaking the **next** upgrade's drift detection (it would mis-read every CORE/PACK region as drifted).

Two changes:
- **Surface + fail closed.** The manifest-spec builder and the `.hos-manifest` assembler now capture stderr; on a crash they print it and `exit 1`. A genuine *regions.py-absent* case still falls back to a WHOLE-only manifest, but now **warns** that region tracking is degraded — distinguished from a *crash* (regions.py present but failing), which refuses to proceed.
- **The version stamp is the commit point.** `.hos-release` is now written **last** — only after the agents *and* the manifest are on disk. Previously it was stamped several steps earlier, so a manifest failure left a new version over a stale/missing manifest. Now any failure in the manifest block precedes the version stamp, so an incomplete install is never marked complete. (Caught by the adversarial review of the first cut of this fix.)

## #218 — runbook only advertises audit queries that work

`docs/OVERSIGHT-RUNBOOK.md` advertised `jq` queries for `panel-run` and `risk-assessment` audit events that **no code emits** (verified: the only emitted events are `gate-auto-reenabled, gate-na, gate-suspended, human-authorization, na-invalidated, step-head, structural-override, suspension-census`). Those queries always returned empty. Replaced with working equivalents (an `escaped-defect` issue query; the `risk-assessment.md` artifact + a `step-head` query) and a note that those agents emit no audit event.

## #223 — runbook gate-suspendability table corrected

The gate-name reference table implied only `security` is suspendable. The evaluator's suspension check is generic over **every** required role; the table now reflects that (with the re-enable nuance: script gates auto-re-enable, reviewer roles + `security` need a human).

### Release integrity note
One installer fix (`#277`, a protected surface) plus two runbook-accuracy fixes, on the `release/v0.3.x` patch line. The `#277` fix was adversarially reviewed (HOS dogfooding) — the review caught a blocking ordering defect in the first cut (`.hos-release` stamped before the manifest `exit 1` could fire), now fixed by making the version stamp the last write. The doc fixes were verify-first (each query/event grep-checked; the gate table cross-checked against the evaluator + suspension template). A third doc issue (#245, deploy-verify) was deliberately **not** changed — it's a contested runtime-capability question routed to pm + architect, not a doc bug. Validated by static check + bash -n. No agent definitions changed; no validation override used.
