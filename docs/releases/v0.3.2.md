# v0.3.2 — the installer never silently swallows a failure

A patch release with two fixes in one theme: `bootstrap/hos_install.sh` must **fail closed and loud** when something under it goes wrong, never report success over a degraded result. No agent, contract, or pipeline changes.

## #276 (high) — a `regions.py` crash no longer silently degrades the install

Several `python3 regions.py …` calls in the region-install path piped stderr to `2>/dev/null` and then **silently fell back** on failure:

| Call | Old silent failure |
|---|---|
| migrate (template) | crash → region left **unwrapped** (composes to nothing) |
| migrate (disk) | crash → silent raw **`cp`** used as the composed "disk" image |
| `base-shas` | crash → silent **`{}`** → plan mis-detects drift |

A crashing subprocess at any of these points produced a wrong-but-silent result — a flat region, a raw copy, or empty base-shas — and the installer still printed **"Done."** For a framework whose whole job is auditable correctness, *a crash that dies silently is the worst failure mode.* (This is also what produced an unattributable "python crash in the terminal" during a real consumer adoption — the traceback was being eaten by `2>/dev/null`.)

**v0.3.2 makes these calls fail closed.** A new `_regions_strict` helper runs `regions.py` capturing stderr; on **any** non-zero exit it **surfaces the traceback and `exit 1`** — and that exit happens in **Phase A (staging)**, *before* Phase B writes any agent, the `.hos-manifest`, or the `.hos-release` stamp. So a crash now aborts with **nothing written, no manifest, no version stamped** — the same fail-closed boundary as the existing layering-drift hard-stop. Two further `regions.py` calls (`inject-pack`, `plan`) already failed closed via the decide-all-then-act abort gate but suppressed their stderr; they now surface it too, so a fail-closed abort is never blind.

Verified: a simulated `regions.py` crash through `_regions_strict` aborts with `exit 1` and the caller never continues onto the degraded data.

## #273 (medium) — `--pr` commit failure no longer pushes an un-committed branch

In the `--pr` path, `git commit … || true` swallowed a commit failure (e.g. a rejecting hook), then pushed the branch and reported "Committed on…". v0.3.2 fails closed on commit failure, skips the push, and reports accurately — the base branch is untouched (the work is isolated on the upgrade branch).

## Upgrading

No action needed for installed projects. The fix changes how a **future** `hos_install.sh` run behaves: if `regions.py` crashes, the run now aborts loudly with the traceback instead of stamping a degraded install.

### Release integrity note
A single-file installer fix on the `release/v0.3.x` patch line; no agent definitions changed. The static consistency check passed, the fail-closed behavior was proven by a direct test (crash → `exit 1`, caller never continues), and the change received a dedicated adversarial control-flow review (HOS dogfooding) which confirmed the `exit 1` precedes any Phase-B write and that no `_regions_strict` call sits inside a command substitution that would trap the exit in a subshell. The review also surfaced — and this release fixes — a robustness gap in the `--pr` return-to-branch step (a failed checkout-back now warns the operator instead of silently leaving them on the upgrade branch); and it flagged two **post-Phase-B** manifest-assembly stderr swallows (same class, deferred-impact) which are filed for a follow-up patch (they do not corrupt the agents this release writes). The cross-vendor agent semantic review was not re-run (no agent definitions changed). No validation override was used. Same fail-closed lesson as v0.3.1 / DECISIONS D46.
