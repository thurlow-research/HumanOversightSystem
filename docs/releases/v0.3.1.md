# v0.3.1 — `--pr` is fail-closed: never silently mutate a consumer's tree

A patch release with a single, **ship-stopper** fix to the project installer. No agent, contract, or pipeline changes — if you are not upgrading a consumer project with `hos_install.sh --pr`, v0.3.1 changes nothing for you.

## The fix (#272)

`hos_install.sh --pr` exists to apply a framework upgrade the **safe** way: on a new branch, opened as a PR, with the consumer's base branch **untouched until they merge**. The bug: when the installer *couldn't* set up that PR, it did the **opposite of what `--pr` promises** — it scaffolded the upgrade **in place**, mutating the consumer's working tree.

The root cause was a guard that looked like a guard but wasn't:

```sh
fail() { err "$*"; ERRORS=$((ERRORS + 1)); }   # prints + counts — does NOT exit
```

So the eligibility check `fail "--pr requested but not possible…"` printed its error and then **execution continued** straight into the in-place scaffolding; the non-zero exit came only at the very end, *after* the files were already written. A second path (branch-creation failure) degraded to in-place explicitly. Either way, a consumer who asked for the reversible path could get their tree modified anyway.

**v0.3.1 makes `--pr` fail-closed.** Any failure to establish the PR branch under an explicit `--pr` — ineligible repo (dirty tree, no `origin`, no `gh`, detached HEAD) or a branch-creation failure — now **hard-stops (`exit 1`) before anything is written**. `--pr` means *PR or nothing*; it never falls back to an in-place upgrade. (Post-scaffold push / PR-create failures, where the work is already safely isolated on the branch and the base is untouched, now also exit non-zero with manual-recovery guidance, instead of reporting false success.)

Verified: `--pr` into a repo with no `origin` exits 1 and leaves the target tree **completely untouched** (zero files written); the happy path is unchanged.

## Why this blocked the line

This surfaced while preparing a real consumer's `--pr` upgrade to v0.3.0. A framework whose one job is **safe, overseen change** cannot ship an installer that quietly does an unsafe, unrequested in-place mutation when the safe path is the one explicitly requested. The lesson — *a fail() that doesn't exit isn't a guard* — is recorded in `research/findings/` and DECISIONS.md.

## Upgrading

No action needed for installed projects. The fix only affects how **future** `hos_install.sh --pr` runs behave. Next time you upgrade a consumer with `--pr`, it will now refuse loudly rather than fall back to in-place.

A single-file installer fix on the `release/v0.3.x` patch line — it touches one script and **no agent definitions**, so validation was scoped to the change (the validation surface should match the change surface, not re-run the full v0.3.0 agent review and manufacture an irrelevant override). The static consistency check passed, and the fix received a dedicated **adversarial control-flow review** (HOS dogfooding) which confirmed: the hard-stop `exit 1` precedes the first filesystem write to the target (line 453), no remaining path lets `--pr` reach in-place scaffolding, and every edge case (detached HEAD, branch collision, dirty tree, no-origin) fails closed. The cross-vendor agent semantic review was **not** re-run (no agent definitions changed). **No validation override was used.** The review also surfaced two pre-existing, out-of-scope installer issues, filed for a later v0.3.x patch.
