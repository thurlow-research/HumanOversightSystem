# v0.3.6 — framework-setup-validator no longer false-fails consumer installs

A single dev-tooling fix.

## #259 — framework-setup-validator stops requiring the unshipped dev-pack validators

`framework-setup-validator`'s `REQUIRED` agent list included the four **framework-dev** validators (`framework-validator`, `framework-setup-validator`, `doc-validator`, `spec-compliance-validator`) — which the framework explicitly does **not** ship to consumers (they validate HOS itself and belong to the hos-dev-pack). So on every *correct* consumer install it reported four false `MISSING` entries and a false **NOT READY** verdict.

Those four are now checked in a dedicated dev-pack block where absence is **INFO**, not `MISSING` — matching the existing treatment of the optional ops/reliability agents. (`deploy-verify` was deliberately left in `REQUIRED`: its status is the separate, still-open #245 — a flagged-missing deploy-verify reflects the real capability question, not a validator bug.)

This agent is framework-dev tooling and is not shipped to consumers, so the fix has no effect on a consumer's runtime — only on HOS-dev validation runs.
