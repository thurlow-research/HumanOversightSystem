# v0.3.4 — honest `--pr` UX: disclosed upgrade PRs, no silent dry-run gap

Two small installer fixes to the `--pr` path. No agent or contract changes.

## #226 — the generated upgrade PR now discloses it's machine-generated

`hos_install.sh --pr` opened a normal, non-draft PR titled `chore(hos): upgrade framework to <ref>` with **no disclosure** that it was machine-generated — a consumer reviewing it couldn't tell at a glance it wasn't hand-authored, which is exactly the transparency the framework requires of AI/automated PRs elsewhere.

The upgrade PR is now:
- **Titled** `🤖 [hos-install] chore(hos): upgrade framework to <ref>`.
- **Bodied** with an explicit disclosure: *generated by `hos_install.sh` (a deterministic installer script — not hand-authored, not LLM-authored); a human must review before merge.*
- **Opened as a draft**, and best-effort labeled `needs-human` (a missing label in the consumer repo never fails the PR).

> Disclosure is framed honestly: the installer is a **deterministic script**, so the PR is disclosed as *automated/tool-generated*, not as `[AI: claude]` — it would be inaccurate to imply an LLM authored content it didn't.

## #274 — `--dry-run --pr` no longer silently drops `--pr`

The PR-setup path is gated on `! $DRY_RUN`, so `--dry-run --pr` previewed an in-place-looking install with no indication that `--pr` was not simulated. The installer now **warns explicitly** that PR-mode isn't simulated in a dry run and that no branch/PR is created — re-run without `--dry-run` to produce the PR.

## Upgrading
No action needed for installed projects. The changes affect how a **future** `hos_install.sh --pr` run presents its PR. Note the behavior change: the upgrade PR now opens as a **draft** — mark it ready when you've reviewed.
