# v0.3.0 — A canonical, layered base-agent team + stack packs

The big one. HOS now **ships its own base-agent team** — the dev team (coder, reviewers, test, design roles) that a consumer previously hand-rolled — as **layered, upgradable agent files**, with **stack-specific depth delivered as packs**. A blank repo gets a structurally complete pipeline; a Django repo gets that pipeline *plus* Django depth, via `--pack django`. (Promised in the v0.2.2 notes; delivered here.)

## What's new

### The base team (16 agents, layered)
HOS installs a canonical 16-agent base team: `pm-agent`, `architect`, `technical-design`, `coder`; the eight reviewers `code` / `security` / `privacy` / `reliability` / `ops` / `ui` / `a11y` / `infra`; `unit-test`, `system-test`; `ops-designer`, `ux-designer`. You no longer write these by hand.

Each agent file is **three regions**:
- **`CORE`** — HOS-owned, stack-neutral, generic best practice (always present).
- **`PACK:<name>`** — HOS-owned, *reusable stack depth* (e.g. `django`), delivered by a pack.
- **`PROJECT`** — **yours**, consumer-owned, **never overwritten** by HOS.

Canonical order is `CORE → PACK → PROJECT`; the most specific layer wins (PROJECT governs over PACK over CORE).

### Pack selection at install
```bash
./hos_install.sh --pack django  /path/to/project     # core + django depth
./hos_install.sh --no-pack      /path/to/project     # bare core (warns: shallow — install a pack before real use)
```
The choice is recorded in `scripts/framework/config.sh` (`PACK="django"`) so upgrades reuse it. Selecting no pack interactively is an error — you must choose `--pack <name>` or deliberately `--no-pack`. Multi-pack (`--pack a --pack b`) is permitted-but-warned (untested in v0.3.0).

### The `django` pack
Ships **12 of the 16** agents with Django depth — `security-reviewer` (CSRF/HTMX, TOTP, tenant-manager bypass, `select_for_update`, ORM injection), `unit-test` (pytest-django, coverage+mutmut, factory_boy, freezegun), `coder`, `code-reviewer`, `system-test`, `a11y-reviewer`, `privacy-reviewer`, `architect`, `technical-design`, `ui-reviewer`, `ux-designer`, `infra-reviewer` (Docker/gunicorn/collectstatic/Caddy/migrate-before-serve). The other 4 (`pm-agent`, `ops-reviewer`, `ops-designer`, `reliability-reviewer`) are CORE-only — deliberately stack-neutral.

Each django-pack agent was **equivalence-verified** against its real-world counterpart: composed `core+django-pack` covers every distinctive Django check the pilot consumer's hand-rolled agent had, with only project-unique content left for PROJECT.

### Upgrades are a real three-way merge
- Your **PROJECT** edits are **always preserved** across upgrades.
- An edited **HOS-owned `CORE`/`PACK`** region **hard-stops the entire upgrade** (exit 4, **nothing written**, no version stamped) and tells you to either move the edit into PROJECT or re-run with `--squash` to take HOS's version. No silent clobber, no half-write.
- A pack switch or `--no-pack` strip drops the old pack's regions cleanly (PROJECT untouched).

Mechanism: one additive `regions.py inject-pack` verb over the existing parse/compose/three-way-merge — no new merge logic, no manifest-schema change.

## ⬆️ For the pilot consumer (and anyone on ≤ v0.2.x)
Adopt v0.3.0 via a **clean install** (not an in-place upgrade), per the layering model:
```bash
./hos_install.sh --release v0.3.0 --pack django  /path/to/your-project
```
You get the full layered team; put your project-specific guidance in each agent's **PROJECT** region (it survives every future upgrade). The CORE/PACK/PROJECT model is documented in `docs/CUSTOMIZATION.md`.

## Verified
Proven end-to-end against the **real installer** (not a simulation) — see `docs/v0.3.0/PACK-INSTALL-VERIFICATION.md`:
