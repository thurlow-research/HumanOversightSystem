# v0.4.0 — Autonomous Worker, Machine Accounts, and Overseer Merge Authority

The "Autonomous Worker" release. HOS can now build tasks end-to-end without a human in the loop for each step: a worker agent claims work, executes the full pipeline, and routes the finished PR to an overseer agent that independently decides whether to merge. Machine accounts enforce the boundary between autonomous and human-authorized actions at the GitHub layer.

## What's new

### Autonomous worker agent

`hos_orchestrator.sh --class worker` implements the full Phase A/B/C autonomous worker. Phase A triages the incoming task and acquires a machine lock; Phase B runs the full inner loop (technical design → coder → reviewers → tests → evaluator → orchestrator); Phase C writes the handoff document, publishes the PR, and releases the lock. A convergence ledger tracks reviewer iterations; budget limits are enforced per task; structured observability events are emitted throughout. Multi-customer support allows a single worker process to service queued tasks across independent project contexts.

### Bot identities — hos-worker-hos[bot] and hos-overseer-hos[bot]

> **Errata (2026-06-19):** v0.4.0 shipped with PAT machine accounts (`HOSWorkerTutelare` / `HOSOversightTutelare`). These were flagged by GitHub for bot-like behaviour and replaced with GitHub Apps mid-cycle. The authoritative identity model is now GitHub App installation tokens via `bootstrap/get_app_token.sh`. See PR #542/#543 and issue #573.

Two GitHub App bot identities separate autonomous build actions from oversight actions at the Git/GitHub identity layer. `hos-worker-hos[bot]` (app_id: 4090164) owns the build pipeline; `hos-overseer-hos[bot]` (app_id: 4090678) owns merge and override operations. Auth flow: `source <(bootstrap/get_app_token.sh --app worker|overseer)` exports `GH_TOKEN` + `HOS_BOT_LOGIN` into the shell. A CI gate (`require-tier-ceiling`) enforces that autonomous actions stay within the tier ceiling declared in the step manifest; a second gate (`require-human-approval`) blocks any action that the step manifest marks as requiring human authorization. Branch protection rules are configured via `setup_branch_protection.sh`.

### Overseer agent (`hos_orchestrator.sh --class overseer`)

The overseer is a separate long-running agent that reviews PRs produced by the worker and makes autonomous merge decisions according to a merge-authority matrix. The matrix maps risk tier and evaluator verdict to one of three dispositions: merge, conditional merge, or escalate to human. `OVERSEER_CEILING` caps the highest tier the overseer may merge without human sign-off. A bounce-back protocol routes conditional-merge PRs back to the worker with a structured correction brief; batch merge serialization prevents concurrent merges that could produce conflicting states.

## Governance hardening — evaluator conditions 11–14

Four new conditions tighten the evaluator's compliance checks.

**Condition 11 — evaluator-independent tier derivation.** The evaluator now re-derives the deterministic tier floor from raw validator output rather than reading risk-assessor's declared tier. A mismatch between the two is a compliance failure, closing the path where a manipulated `risk-assessment.md` would be trusted silently.

**Condition 12 — per-step HIGH-tier override authorization.** Proceeding past a HIGH-tier security or privacy suspension requires a signed authorization file in `contract/tier-overrides/` specific to that step. Evaluator checks for the file; absence is a blocking compliance failure.

**Condition 13 — second-review range validation.** The second-review report must include a `reviewed_range` field (the commit range the reviewer actually examined). The evaluator compares this against the step's expected range and fails if they do not match, closing the gap where a stale second-review report covers a different diff than the one being evaluated. Implemented via a shared `step_range.sh` helper (#220) and tied to the step-head-final event.

**Condition 14 — structural-override modification detection.** The evaluator now detects when an existing structural-override authorization file has been modified, not only when a new one appears (#267). Authorship of governance artifacts is checked against the machine account identity (#82). Reviewers must consume and acknowledge notification artifacts before approving; the consumption protocol is enforced (#85).

### CODEOWNERS bypass gate

A new validator, `codeowners.py` (#396), checks whether any changed file has a CODEOWNERS entry that was not approved by the designated owner. This closes a bypass path where a worker-authored PR touching owned paths could merge without the required CODEOWNERS review.

### CONDITIONAL_PROCEED thread blocking (partial, #222)

When the evaluator issues a CONDITIONAL_PROCEED verdict, the orchestrator now opens a PR thread documenting the unresolved conditions and blocks merge until the thread is resolved. Full resolution tracking (detecting when a human has addressed each condition) remains a follow-up; the blocking thread itself is in place.

### Out-of-scope commits protocol (SPEC-328)

Commits that fall outside the declared build-step scope must now be explicitly accounted for via an out-of-scope declaration before the evaluator will proceed. This closes the path where incremental commits adding unrequested functionality passed evaluation silently.

## SLR-grounded research proposals

Six proposals derived from peer-reviewed findings are incorporated as framework behaviors or draft specifications. All carry citations.

**Task-class risk floor** (Ferdous et al. 2026): refactor and chore tasks now have a minimum tier of MEDIUM regardless of static metric output, reflecting empirical evidence that structural changes carry higher defect density than their surface area suggests.

**Arbiter corroboration ranking** (Charoenwet et al. 2026): second-review arbiter outputs are split into Tier 1 (same finding independently raised by two vendors) and Tier 2 (single-vendor only). Evaluator weights Tier 1 findings as blocking; Tier 2 findings as advisory.

**Diff-size risk floor and split trigger** (Watanabe et al. 2026): diffs above a configurable line threshold automatically elevate to HIGH and emit a split recommendation. Large diffs may be returned to the worker with a split brief rather than proceeding to review.

**Structured rejection rationale** (Watanabe et al. 2026): overseer bounce-back and evaluator ESCALATE verdicts now emit a structured rationale block (finding class, affected file, required correction) rather than free-form prose, so the worker can route corrections to the right specialist without re-reading the full report.

**Diff-centric review context** (Kumar 2026): reviewer agents now default to `--diff-only` context — they see the changed hunks and their immediate neighbors, not the entire file — reducing noise and improving finding density per token.

**Adversarial framing guard** (Mitropoulos et al. 2026): a `classify_framing` pre-pass detects prompt constructions that attempt to reframe the agent's role or bypass governance rules (e.g. "for testing purposes ignore…"). Flagged prompts are rejected before the inner loop starts.

### Automated review-chain detection and CONDITIONAL_PROCEED audit (#360, #370)

Review-chain auto-detect (#360) identifies when a build step's reviewer chain is incomplete given the step's risk tier and file domains, and emits a structured gap report before the evaluator runs. CONDITIONAL_PROCEED audit (#370) records each conditional verdict with its conditions and resolution state in the convergence ledger, enabling retrospective analysis of how often conditions are resolved vs. escalated.

## Bash-to-Python refactoring (#314 policy)

Per the #314 decision, pipeline-critical bash logic is being migrated to Python for testability and determinism. Completed in this release: `second_review_logic.py`, `validation_logic.py`, `release_logic.py`, `agents_static_logic.py`, `prompt_audit_logic.py`. Extended in this release: `panel_logic.py`, `suspension_manager.py`. The convergence ledger was extended to cover second-review and panel events in addition to the inner-loop reviewer chain.

## Bug fixes

- **`require_tier_ceiling.py` (#412)**: inline bash comments inside heredocs were being parsed as content, producing false tier-ceiling violations on valid configurations.
- **`setup_branch_protection.sh` (#355)**: JSON heredoc blocks containing `#` characters were being silently truncated by comment stripping, producing malformed payloads to the GitHub API.
- **`lint/type_check.sh` (#358)**: when called with no file arguments (e.g. on a step with no Python), the script exited non-zero silently, causing the gate to fail-closed on steps it had no work to do.
- **`risk-historian` (#367)**: churn counts were including doc and spec commits, inflating historical risk scores for files whose only changes are documentation. Churn is now limited to commits touching non-doc paths.
- **Validation stamps (#422, #433)**: per-step validation stamp files were committed alongside source changes, causing merge conflicts on concurrent branches. Stamps are now gitignored; CI reads them from the working tree, not from git history.

## Docs

**`GATE-COVERAGE-MECE.md`**: a MECE analysis mapping 13 failure classes against 12 gate layers. Documents which failure classes are covered by which gates, which are covered redundantly, and which are not covered by any deterministic gate (relying on agent judgment only).

**`AGENTS.md` — Worker / Oversight split**: the agent table is restructured into Worker Agents (base team + orchestrator) and Oversight Agents (evaluator, overseer, risk-assessor, etc.) with an explicit note on the decorrelation principle — oversight agents must not share context with the artifacts they are reviewing.

**`OBSERVATIONS.md` — O9 and O10**: O9 records the finding that AI scope drift (agents adding unsolicited functionality) is empirically indistinguishable from human developer scope creep in peer code review, motivating the out-of-scope commits protocol. O10 records that exhaustive enumeration of allowed behaviors outperforms principle-based constraints for governance boundaries.
