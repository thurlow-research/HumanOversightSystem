# v0.5.0 — Governance, Accuracy & Usability: Release Plan

**Theme:** Tighten governance controls, improve accuracy of oversight signals, and fix usability gaps that surfaced during v0.4.0 and v0.4.1 operation.  
**Status:** Active (70 of ~74 issues closed — winding down)  
**Milestone:** [v0.5.0 — Governance, Accuracy & Usability](https://github.com/thurlow-research/HumanOversightSystem/milestone/8)  
**Promoted from:** v0.4.2 (cascade: v0.4.2 → v0.5.0 when v0.4.1 scope expanded; infrastructure reliability and governance hardening items from the old v0.5.0 Quality plan pulled into this release)

---

## Theme rationale

v0.4.0 shipped an autonomous loop. v0.4.1 stabilized it. v0.5.0 makes it trustworthy. Three failure modes from field operation drove this theme:

1. **Governance gaps** — merge-authority decisions were under-specified, allowing edge cases to slip through without proper authorization. Audit log coverage was incomplete.
2. **Accuracy gaps** — infrastructure reliability issues (auth gaps, GraphQL quota failures, stale-claim detection) caused the pipeline to produce wrong answers under load.
3. **Usability gaps** — blocking issues in the worker loop spec, untrusted-input handling, stamp merge conflicts, and config resolution impeded productive use.

This release resolves all three before v0.6.0 adds quality measurement on top of a system that must first be correct.

## Buckets

### Bucket 1 — Governance Hardening

**Why:** The oversight pipeline must have unambiguous authority rules and complete audit coverage before quality measurement can be trusted.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Merge-authority guards | Deterministic `decide_merge_authority()` function; guards fire in correct order (#756/#761 before #589) | #756, #761 |
| Overseer auto-merge after human approval | Overseer detects human GitHub approval and merges autonomously; removes second manual click | #589 |
| Audit-log migration | Phased migration of audit events to structured schema (P0–P5) | #888 |
| Untrusted input security | Issue/PR content classified as untrusted DATA; prompt-injection guard enforced in worker loop | #734 |
| Overseer reads validator artifacts | Overseer verifies artifact content, not just CI pass/fail | #555 |
| Ratchet principle for test failures | Policy: touching a test file owns making the whole module green | #580 |

### Bucket 2 — Infrastructure Reliability (Accuracy)

**Why:** v0.4.0 exposed auth gaps, GraphQL quota failures, and a stamp mechanism that blocked 15+ PRs. Accuracy of oversight decisions depends on a reliable API substrate.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Ban GraphQL; map to REST | Forbid `gh pr list`, `gh issue list`, `gh pr view --json`; document REST equivalents | #571 |
| Full GitHub App migration | Remaining callers: `setup_clis.sh`, decommission PAT accounts | #572 |
| GitHub API robustness | ETag-based read cache + idempotent write queue; graceful rate-limit degradation | #570 |
| Stale-claim detection | Pre-PR guard blocks pushes overlapping commits already in an open PR | #754 |

### Bucket 3 — Usability

**Why:** If the pipeline is accurate but hard to use, humans route around it.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Stamp redesign | Content-hash anchored stamps (filename = SHA256 of validated agent files); eliminates merge conflict cascade | #552 |
| Config file hierarchy | Canonical `~/Code/<Project>/.config/hos/` layout; 3-level path resolution in `get_app_token.sh` | #606 |
| Triage agents | Haiku 4.5 gateway agents (pr-triage, arch-triage, tech-design-triage); mechanical pipeline enforcement | #558 |
