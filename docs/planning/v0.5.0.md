# v0.5.0 — Quality: Release Plan

**Theme:** Extend HOS from a governance framework that enforces rules to one that measures and improves quality over time.  
**Status:** Planning  
**Milestone:** [v0.5.0 — Quality](https://github.com/thurlow-research/HumanOversightSystem/milestone/2)  
**Supersedes:** `Main/Main/docs/v0.5.0-plan.md` (moved here 2026-06-19)

---

## Theme rationale

v0.4.0 makes the loop autonomous. v0.5.0 makes it *self-improving*. Two sources of ideas: a systematic literature review (Cobb & Mills 1990 on statistical quality control; Poppendieck & Cusumano 2012 on lean) identified capabilities that flow naturally from HOS's architecture; and v0.4.0 field operation surfaced infrastructure reliability gaps. Both deserve to land before v0.6.0 asks for higher velocity.

---

## Buckets

### Bucket 1 — SLR-Derived Quality Capabilities

**Why:** The literature gives us validated quality models; HOS's pipeline structure makes them implementable cheaply.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| MTTF/B-factor certification | B = MTTF_{n+1}/MTTF_n as consumer PR gate; release blocks if B < 1 | #563 |
| Usage-profile-weighted tests | Test distribution weighted by operational usage profile (Adams study: 21× more effective than coverage-first) | #564 |
| Lean waste report | Classify audit log cycle time into lean waste categories at each release | #565 |
| Quality ratchet | Defect density monotonically non-increasing across releases | #566 |
| Certification independence | System-test agent derives tests from spec, not from coder's coverage map | #568 |
| WIP limits *(deferred v0.6.0)* | Consumer-configurable max PRs in flight | #567 |
| B-factor hard merge gate *(deferred v0.6.0)* | B < 1 blocks merge, not just release | #569 |

### Bucket 2 — Infrastructure Reliability

**Why:** v0.4.0 exposed auth gaps, GraphQL quota failures, and a stamp mechanism that blocked 15+ PRs.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Ban GraphQL; map to REST | Forbid `gh pr list`, `gh issue list`, `gh pr view --json`; document REST equivalents | #571 |
| Full GitHub App migration | Remaining callers: `setup_clis.sh`, decommission PAT accounts | #572 |
| GitHub API robustness | ETag-based read cache + idempotent write queue; graceful rate-limit degradation | #570 |
| Stamp redesign | Content-hash anchored stamps (filename = SHA256 of validated agent files); eliminates merge conflict cascade | #552 |
| Config file hierarchy | Canonical `~/Code/<Project>/.config/hos/` layout; 3-level path resolution in `get_app_token.sh` | #606 |
| Triage agents | Haiku 4.5 gateway agents (pr-triage, arch-triage, tech-design-triage); mechanical pipeline enforcement | #558 |

### Bucket 3 — Governance Hardening

**Why:** Known compliance gaps from v0.4.0 that were deferred or blocked on human decisions.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Overseer reads validator artifacts | Overseer verifies artifact content, not just CI pass/fail — blocked on #562 (human decision) | #555 |
| Overseer auto-merge after human approval | Overseer detects human GitHub approval and merges autonomously; removes second manual click | #589 |
| Ratchet principle for test failures | Policy: touching a test file owns making the whole module green | #580 |
| Worker loop spec improvements | Additional loop-start precheck improvements discovered in v0.4.0 | #550, #551 (done in v0.4.0) |
