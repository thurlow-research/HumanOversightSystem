# v0.7.0 — Agility: Release Plan

**Theme:** Fully embrace agile — faster feedback loops, pull-based flow, continuous self-improvement, and reduced friction at every touchpoint.  
**Status:** Planning (early)  
**Milestone:** [v0.7.0 — Agility](https://github.com/thurlow-research/HumanOversightSystem/milestone/3)  
**Renumbered from:** v0.6.0 — Agility (cascade: v0.4.2 promoted to v0.5.0, shifting this release from v0.6.0 → v0.7.0)

---

## Theme rationale

v0.6.0 teaches HOS to measure quality. v0.7.0 teaches it to move faster *because of* that measurement. Agility in software development isn't about cutting corners — it's about shortening feedback loops, eliminating waste, and letting teams pull work rather than having it pushed at them. HOS already enforces discipline; v0.7.0 makes that discipline feel lightweight. The result: a team using HOS should feel more productive, not more constrained.

The five core agile values as they apply to HOS:
1. **Shorter cycles** — reduce PR batch size, minimize human wait time
2. **Pull system** — WIP limits prevent overloading the review queue
3. **Continuous improvement** — quality metrics drive the next sprint
4. **Feedback fast** — reactive pipeline, instant detection of regressions
5. **Dogfood** — use HOS to build HOS; find pain points from the inside

## Buckets

### Bucket 1 — Flow: Pull System & WIP Limits

**Why:** Agile kanban thinking — work should be pulled by capacity, not pushed by demand. Unconstrained WIP causes queues, delays, and context-switching overhead.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Configurable WIP limits | Consumer sets max PRs in flight via step-manifest; overseer enforces | #567 |
| B-factor as hard merge gate | B < 1 (reliability regressing) blocks merge, not just release | #569 |
| Overseer batch merge serialization | Already in v0.4.0; ensure it handles WIP limits correctly | (existing) |

### Bucket 2 — Continuous Improvement: Close the Feedback Loop

**Why:** Agile teams retrospect after every sprint. HOS needs machine-readable quality trends so the retrospective data is automatic.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Dogfood: use HOS to build HOS | Run the full HOS pipeline on HOS development; find pain points from the inside | #176 |
| HOS vs. SWE best practices audit | Systematic comparison: where does HOS diverge from established practice and why? | #190 |
| agy cross-vendor review of release scripts | Independent cross-vendor validation of release automation | #91 |
| Reactive pipeline / Jidoka | Stop-on-defect: pipeline halts on first escape rather than continuing to completion | #53 |

### Bucket 3 — Developer Experience: Remove Friction

**Why:** An agile team should be able to respond to change quickly. HOS should get out of the way when it can, and be clear about why it can't when it must.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Pack auto-detection | Detect stack from project contents → auto-select the right HOS pack | #239 |
| Generalize local fixes | Consumer-local overrides without forking the framework | #66 |
| Override stamp restoration | Restore main's #269 override stamp (clobbered by erroneous FF) | #279 |

### Bucket 4 — Process: Agile Governance

**Why:** The governance model itself should be agile — adapting to what the team learned, not frozen at design time.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| Human role optimization | Reduce unnecessary human touchpoints; clarify which decisions only humans can make | #67 |
| Cursor integration | Explore Cursor as a worker/reviewer runtime (operator preference) | #268 |
