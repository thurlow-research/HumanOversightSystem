# v0.6.0 — Quality: Release Plan

**Theme:** Extend HOS from a governance framework that enforces rules to one that measures and improves quality over time.  
**Status:** Planning  
**Milestone:** [v0.6.0 — Quality](https://github.com/thurlow-research/HumanOversightSystem/milestone/2)  
**Renumbered from:** v0.5.0 — Quality (cascade: v0.4.2 promoted to v0.5.0, shifting this release from v0.5.0 → v0.6.0)

---

## Theme rationale

v0.5.0 makes HOS trustworthy. v0.6.0 makes it *self-improving*. Two sources of ideas: a systematic literature review (Cobb & Mills 1990 on statistical quality control; Poppendieck & Cusumano 2012 on lean) identified capabilities that flow naturally from HOS's architecture; and v0.4.0 field operation surfaced infrastructure reliability gaps resolved in v0.5.0. Both sets of insights land here once the v0.5.0 substrate is stable.

---

## Buckets

### Bucket 1 — SLR-Derived Quality Capabilities

**Why:** The literature gives us validated quality models; HOS's pipeline structure makes them implementable cheaply.

| Feature | Description | Issue(s) |
|---------|-------------|---------|
| MTTF/B-factor certification | B = MTTF_{n+1}/MTTF_n as consumer PR gate; release blocks if B < 1 | #563 |
| Usage-profile-weighted tests | Test distribution weighted by operational usage profile (Adams study: 21× more effective than coverage-first) | #564 |
| Lean waste report | Classify audit log cycle time into lean waste categories at each release | #565 |
| Quality ratchet | Defect density monotonically non-increasing across releases | #566 |
| Certification independence | System-test agent derives tests from spec, not from coder's coverage map | #568 |
| WIP limits *(deferred v0.7.0)* | Consumer-configurable max PRs in flight | #567 |
| B-factor hard merge gate *(deferred v0.7.0)* | B < 1 blocks merge, not just release | #569 |
