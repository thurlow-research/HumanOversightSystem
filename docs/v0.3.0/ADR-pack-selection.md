# ADR-031 — Pack install/selection mechanism (HOS v0.3.0)

**Status:** ACCEPTED — binding. *Implements* spec §11a S1/S2 and resolves `TECHNICAL-DESIGN.md` Flags #1 (pack selection) and #2 (multi-pack guard/permit). Binds `coder` (authoring `packs/django/`, wiring `--pack`), `technical-design` (§7.1 blank-install path), and `code-reviewer`.

**Context.** The region model (`TECHNICAL-DESIGN.md` §2–§5, `regions.py`) is complete and presumes a *selected pack* whose `PACK:<name>` region is already present in each staged template. Neither the spec nor TD defined *how* the pack is chosen, where its content lives on disk, how it is composed into the staged template before `regions.py plan`, where the choice is recorded for upgrade reuse, or how `--no-pack` reconciles with #237. The human settled selection UX (S1/S2); this ADR locks the *mechanism* around it. **Constraint:** reuse `compose`/`plan_upgrade`/`manifest_rows`/`assemble_manifest` and the existing Phase-A/B flow. Do **not** invent a parallel path. `regions.py` changes are flagged explicitly at the end (there is exactly one, additive).

---

## Decision 1 — Selection mechanism, recording, and upgrade read-path

**1.1 Selection (locks S1).** Pack is chosen by, in precedence order:
1. `--pack <name>` flag on `hos_install.sh` (highest — overrides config),
2. `PACK=<name>` key in the target's `scripts/framework/config.sh`,
3. neither present → the **no-pack decision** (1.3).

`<name>` must match the existing pack-slug grammar `[a-z0-9][a-z0-9-]*` (already enforced in `regions.py` `_PACK_SLUG`). The installer validates the slug and that `packs/<name>/` exists in the HOS source; an unknown pack is a **hard error** (exit non-zero, nothing written) — fail-closed, never silently fall through to core-only.

**1.2 Recording (the upgrade read-path).** The resolved pack name is **written to `config.sh` as `PACK="<name>"`** on every install where a pack is selected (append-if-absent, overwrite-if-`--pack`-given-and-differs). `config.sh` is the **single record of the choice** — it already survives upgrades (consumer-owned, never overwritten wholesale; the installer only appends keys). On a subsequent upgrade with no `--pack` flag, step 1.1.2 reads `PACK` back from `config.sh` and reuses it. **No new state file.** `.hos-manifest` records *what was composed* (the `PACK:<name>` rows — Decision 3); `config.sh` records *the intent* so an upgrade re-derives the same composition. The two are independently sufficient to detect a pack switch (manifest says pack A, config says pack B → 3.4).

> **Coder note:** `config.sh` is generated by `scripts/framework/install.sh`. Add a `PACK="${NEW_PACK}"` line to its heredoc (after `PROJECT_STACK`) and a `--pack`/prompt path in that tool. But `hos_install.sh` must **not** depend on install.sh having run — it appends `PACK="<name>"` itself (mirroring the existing placeholder append-loop) when `--pack` is given and the key is absent. One append helper, reused.

**1.3 The no-pack decision (reconciles S1 "error" with #237 "warn — core is real").** Split by path:
- **No pack resolvable AND interactive (`-t 0`) AND no `--no-pack`:** the install is **incomplete by intent** → S1 **error** path. Exit non-zero: `no PACK selected — pass --pack <name> (e.g. --pack django), set PACK= in config.sh, or pass --no-pack to install the bare core deliberately`. Do **not** install a half-configured project silently.
- **`--no-pack` given (explicit opt-out):** proceed with **core only**, and **WARN** per #237: `installing bare core with no pack — core enforces generic best practices but is shallow; install a pack before first real use`. The bare core *is* a real, working install (CORE + empty PROJECT); the consumer deliberately declined stack depth.
- **Non-interactive / CI with no pack and no `--no-pack`:** treat as the error path (exit non-zero) — CI must be explicit. (`--no-pack` is the CI escape hatch for a deliberate core-only build.)

This gives S1 its hard default (you don't *accidentally* ship core-only) and #237 its truth (core-only is a *valid, supported, warned* configuration when chosen).

**New flags on `hos_install.sh`:** `--pack <name>` and `--no-pack` (mutually exclusive; both-set → usage error). `--pack=<name>` form too, matching the existing `--release=*` convention.

## Decision 2 — On-disk pack layout (the `packs/` contract)

**2.1 Directory structure.** Packs live in the HOS repo at `packs/<name>/`, **one region-body file per agent the pack deepens**:

```
packs/
  django/
    pack.toml                     # pack metadata (name, description, version, requires)
    security-reviewer.md          # the PACK:django region BODY for security-reviewer
    reliability-reviewer.md       # …only agents the pack actually deepens get a file
    dep-mapper.md
    ...                           # agents with NO django depth have NO file here
  hos-dev/
    pack.toml
    post-change-sweep.md
    ...
```

**2.2 Mapping rule (file → region).** For pack `<name>`, the file `packs/<name>/<agent>.md` contains **exactly the region body** that becomes the `PACK:<name>` region of `.claude/agents/<agent>.md`. It is **raw body bytes only — no markers, no front-matter**. The installer wraps it in `<!-- HOS:PACK:<name>:START -->`…`:END -->` via `compose`. An agent with **no file** in `packs/<name>/` gets **no PACK region** for that pack — the absence is the signal ("this agent has no stack depth here"). `ls packs/django/` *is* the list of agents django deepens.

**2.3 Why region-body files (not one monolith, not whole-agent files).**
- *(a) one `packs/<name>/pack.md` with all regions* — rejected: forces a second in-file parse format and couples unrelated agents (a django security edit churns the whole pack's review surface).
- *(b) full agent `.md` files carrying their own `PACK:<name>` markers* — rejected: duplicates the CORE body into the pack tree (drift between the pack's copy and `.claude/agents/<agent>.md`'s CORE) and requires the installer to *extract* the PACK region before composing. Violates "one source per region."
- **(c, CHOSEN) one body-only file per deepened agent** — the pack owns *only* its region's bytes; CORE stays solely in `.claude/agents/<agent>.md`; `compose` is the only thing that co-locates them. No region body is ever duplicated on disk. Minimal unit mapping 1:1 onto `Region(id="PACK:<name>", body=<file bytes>)`.

**2.4 `pack.toml` (metadata, minimal).** v0.3.0 keys: `name` (must equal the dir name — validated), `description`, `version` (semver string; informational for now), `requires` (optional list of pack slugs; **unused** beyond being recorded — multi-pack untested, Decision 4). The installer reads only `name` (sanity-check vs dir) in v0.3.0; the rest is forward-seam (community registry later). Trivial parse — `grep`/`cut` for `name`, no TOML lib dependency (consistent with `regions.py`'s no-third-party-dep posture).

**2.5 Placeholder rule (binds D1a/D7).** Pack body files are **PACK regions** → they **MUST NOT contain install-time placeholders** (D1a). The hos-dev CI path runs `regions.py validate --placeholder-keys <csv>` over each *composed* agent (D7) and hard-fails `E_PLACEHOLDER_IN_CORE_PACK` if a `{KEY}` leaks in. Authoring guidance for `packs/<name>/*.md`: use runtime self-direction, never `{SPEC_FILE}`-style tokens.

## Decision 3 — Compose/stage flow (where the pack enters Phase A)

The pack is composed into the **staged template** at step **A1**, *after* substitution-staging and *before* `regions.py plan`. Single integration point. The on-disk file, `plan_upgrade`, the manifest, and the merge are all unchanged downstream — they simply see a template that now carries a `PACK:<name>` region.

**3.1 Staged-template assembly (replaces the current "wrap flat → CORE" stub).** For each consumer agent, in Phase A1:

```
1. cp $HOS_SOURCE/.claude/agents/<agent>.md  → $stage          # CORE template (flat pre-Phase-2, or markered)
2. perl -i substitute $stage  using target config.sh           # D6 — the ONLY substitution engine (unchanged)
3. if $stage has no markers:  regions.py migrate $stage --ships yes  → $stage   # flat→CORE (existing forward-compat)
4. IF a pack is selected AND packs/<pack>/<agent>.md exists:
       compose CORE($stage) + PACK:<pack>(body=packs/<pack>/<agent>.md bytes)  → $stage
   (no pack file for this agent → leave $stage as CORE-only)
5. regions.py plan $disk $stage --base-shas … [--first-install]  (UNCHANGED)
```

**3.2 How step 4 composes — reuse, don't reinvent.** The pack body is **not** substituted (D6 — PACK is placeholder-free). The clean way: a **new thin CLI verb** `regions.py inject-pack <staged-template> --name <pack> --body-file <packs/<name>/<agent>.md>` that `parse`s the staged template, appends `Region(id="PACK:<name>", name=<name>, body=<body bytes>)`, and prints `compose(...)`. ~15 lines, pure-function-backed (`parse` + list-append + `compose`, all existing), keeping *all* region writing inside the one writer. Multi-pack falls out free: call once per pack; `compose`'s `_canonical_order_key` alpha-orders them.

  **Rejected:** bash concatenating marker lines + body — re-implements `compose`'s marker emission and `_normalize_body` in bash, diverging from the sha `region_sha` computes and breaking the D1/§2.6 disk==manifest identity. The pack body **must** flow through `compose`/`_normalize_body` so its on-disk bytes equal its manifest sha. Non-negotiable.

**3.3 `base_sha` recording for the PACK region.** After step 4 the staged template has a `PACK:<name>` region. From there, the *existing* path:
- `plan_upgrade` computes `incoming = region_sha(pack body)` for the PACK region (`merge_region` handles any non-PROJECT id).
- On **blank install** (§7.1): PACK region freshly introduced (absent on disk) → `disk_sha is None` → `REFRESH` → write body, **stamp `base_sha = incoming`** in the manifest (already implemented). The row `<.claude/agents/<agent>.md>\tPACK:<name>\t<sha>` is emitted by `_plan_manifest_rows` with **no change**.
- **No special-casing.** PACK's `base_sha` is recorded by the identical mechanism as CORE's.

**3.4 Upgrade path (pack version bump → REFRESH vs consumer-edited-PACK HARD-STOP, binds D2).**
- Installer resolves the pack from `config.sh` `PACK=` (1.2), composes the **new** pack body into the staged template (step 4).
- Per agent's `PACK:<name>` region, the existing three-way fires:
  - `base == disk` (untouched) ∧ `disk != incoming` (HOS bumped) → **REFRESH** — version bump lands cleanly, `base_sha` re-stamped.
  - `base != disk` (consumer **edited**) ∧ `disk != incoming` → **HARDSTOP** unless `--squash` (D2). Refuse upgrade, per-region drift report names the `PACK:<name>` region, remedies = `--squash` or move the edit to PROJECT. **No new code** — `merge_region` is region-id-agnostic.
  - Convergent edit (`disk == incoming`) → KEEP/realign (row 3).
- **Pack switch (config says django, manifest has `PACK:flask` rows):** the old pack's region is now absent from the staged template → the **removed-region sweep** (D9): `PACK:flask` in `base_shas` but not in template → `removed=True` → unedited ⇒ **DROP**, edited ⇒ **HARDSTOP** unless `--squash`/`--prune`. **Zero new merge logic** — a pack switch is exactly "HOS retired these regions" from the manifest's view.

## Decision 4 — Multi-pack: permit-but-warn (locks S2, resolves TD Flag #2)

**Confirmed: permit, do not guard.** Multiple packs (`--pack a --pack b`, or future `PACK="a b"`) → compose each via repeated `inject-pack`. `compose`'s `_canonical_order_key` is the **composition contract**: PACK regions ordered alphabetically by name between CORE and PROJECT. Already implemented + tested; no change.

**Warning fires** in `hos_install.sh` at pack resolution, **before** Phase A, once per install: `multiple packs selected (a, b) — multi-pack composition is UNTESTED in v0.3.0 (alphabetical order, no conflict resolution); single-pack is the supported path`. One warning, not per-agent. **No hard refusal.** `--pack` accepts repetition (append to a `_packs` array); a future `PACK="a b"` space-split is the seam, but **v0.3.0 wires single-value `PACK=` and repeated `--pack`** — don't over-build the untested path.

> `regions.py` already refuses *duplicate* pack names (`E_DUP_PACK`) and orders distinct packs alphabetically — format-level integrity of multi-pack is guaranteed; the only "untested" part is *semantic* conflict between two packs' instructions to the same agent (a content concern, not a mechanism one).

## Decision 5 — hos-dev-pack symmetry (confirmed)

The mechanism expresses "core + hos-dev-pack" with **zero special-casing**. "Building HOS = core + hos-dev-pack" becomes literally:

```
./bootstrap/hos_install.sh --local --pack hos-dev <hos-repo-itself>
```

`packs/hos-dev/` holds body files for agents the framework-dev validators deepen (`post-change-sweep.md` gaining its framework-review track, etc.). The 4 framework-dev validators (`framework-validator`, `doc-validator`, `spec-compliance-validator`, `framework-setup-validator`) are **not** in `consumer_agents.txt` (dev-only), so they're **not** installed by the consumer copy-loop — but the hos-dev-pack's *region bodies* (deepening shipped agents like `post-change-sweep`) compose through the identical path.

**Clarification for the coder:** an agent that exists *only* in hos-dev (the 4 dev-only validators) is a separate concern from *pack region bodies*. Packs deepen **shipped** agents via region bodies; they do not introduce net-new agent *files*. The 4 dev-only validators remain whole-file agents gated by `dev_only_agents.txt`, installed only in the HOS-self / hos-dev-pack context — the existing completeness-gate sub-case A concern (TD §6.2), **not** the pack-region mechanism. Do not conflate "pack" (region depth on shipped agents) with "dev-only agent set" (which agent *files* ship).

## `regions.py` delta (the only flagged change)

**One additive CLI verb, no new logic, no change to any existing function or the three-way merge:**

```
regions.py inject-pack <staged-template-file> --name <pack> --body-file <path> [--in-place]
  → parse(staged-template)
  → append Region(id=f"PACK:{name}", name=name, body=read(body-file), start_line=0, end_line=0)
  → print compose(ParsedAgent(front_matter=..., regions=[...existing..., pack_region]))
```

Reuses `parse`, `Region`, `compose` (already alpha-orders + normalizes via `_normalize_body`), and the `--in-place`/stdout convention of the other write-capable verbs. It must `validate()` the result and exit `EXIT_INVALID` on failure (e.g. injecting a duplicate pack name → `E_DUP_PACK`). **Everything else uses `regions.py` as it stands** — `plan_upgrade`, `merge_region`, `manifest_rows`, `_plan_manifest_rows`, `assemble_manifest`, the removed-region sweep, and `_canonical_order_key` all already handle `PACK:<name>` as a first-class HOS-owned region. No schema change, no migration change, no manifest-format change.

## Install control-flow delta (for the coder, against current `hos_install.sh`)

1. **Args:** add `--pack)` (append to `_packs=()`), `--pack=*`, `--no-pack)` (`NO_PACK=true`). Mutually-exclusive check (`--no-pack` + non-empty `_packs`) → usage error.
2. **Pack resolution (new block, before the agent copy-loop):** resolve `_packs` from flags else `PACK=` in `config.sh`; empty + `NO_PACK` → core-only + #237 WARN; empty + interactive + not `NO_PACK` → S1 ERROR exit; validate each `packs/<name>/` exists (unknown → hard error); `>1` pack → multi-pack WARN. Append `PACK="<name>"` to `config.sh` if a single pack resolved from a flag and the key is absent/differs.
3. **A1 staging (replace the flat-wrap stub):** after substitution + flat-wrap, for each selected pack with a `packs/<pack>/<agent>.md` present, call `regions.py inject-pack $stage --name <pack> --body-file packs/<pack>/<agent>.md --in-place`. Injection order irrelevant (`compose` re-sorts alpha).
4. **Phase A/B, manifest, release, drift, removed-sweep:** **unchanged.**
5. **`scripts/framework/install.sh`:** add `PACK="${NEW_PACK}"` to the config.sh heredoc and a `--pack`/prompt path.

## What this does NOT change (guardrails)

- `.hos-manifest` schema (still `path\tregion\tsha`, schema v2; PACK rows already specified, TD §1.2).
- The three-way merge, `--squash`, `--prune`, the removed-region sweep, drift hard-stop — all region-id-agnostic, all already cover PACK.
- The PROJECT never-written invariant (§4.4) — packs touch only PACK regions; PROJECT untouched. The separate post-core project-install step (spec §10.4 "borg") populates PROJECT and is **out of scope of this ADR**.
- Substitution boundary (D6) — perl runs over CORE in the staged template *before* `inject-pack`; PACK bodies are placeholder-free and never substituted.

---

## Open seams noted, not built (per spec §10.2/§10.5)

- **Fetched/distributable packs** — `packs/<name>/` is in-repo for the pilot; `pack.toml`'s `version`/`requires` are the forward seam for a fetched artifact + registry later. v0.3.0 reads only `name`.
- **`PACK="a b"` multi-value in config.sh** — only single-value `PACK=` + repeated `--pack` wired in v0.3.0.
- **Pack auto-detection** — out per the human's constraint (→ #239). Selection is explicit.
