# v0.3.0 Spec — Canonical Base Agents (core / stack-pack / project)

**Status:** spec for review. Design only — no implementation until reviewed. Ships in **v0.3.0** alongside machine accounts (#152). Tracking issue: #228.

---

## 1. Problem & vision

HOS today ships the **oversight layer** (risk-assessor, evaluator, the reviewers it invented) but **not the base/worker team** (coder, code-reviewer, security-reviewer, architect, …). Those were born in the CPS pilot, hand-authored and Django-specific, and never canonicalized back into HOS. Consequences:

- A **blank repo install gives no working team** — only the 12 oversight agents; the consumer must hand-author coder/reviewers (which CPS did, then hacked around our bugs).
- HOS's expectations of those roles are **implicit and scattered** (contract `role_mappings`, the role-responsibility table, `base-agent-register-examples.md`) — never stated as one canonical position.
- There is **no upgrade path** for base agents: a consumer's bug-workarounds can't be cleaned up, and improvements can't be pushed.

**Vision:** HOS ships and owns a **canonical base team**. A blank repo gets a complete, working, *upgradable* pipeline. Domain specifics live in a layer HOS never overwrites. The same mechanism lets HOS **dogfood its own pipeline** to build itself.

---

## 2. The canonical role set (the "reset our vision")

HOS expects these roles to exist and defines each one's **core responsibility** (generic, stack-neutral). This formalizes the implicit role-mapping table.

| Role | Core responsibility (generic) | Class |
|---|---|---|
| `coder` | Implement to the technical design; iterate with code-reviewer | build |
| `architect` | System design, ADRs, cross-cutting technical decisions | build |
| `pm-agent` | Requirements, spec ownership, test-plan sign-off | build |
| `technical-design` | Translate ADR → implementation contract; own spec-gap routing | build |
| `code-reviewer` | Correctness, design adherence, idioms | review |
| `security-reviewer` | Auth bypass, injection, authz, session, OWASP | review |
| `privacy-reviewer` | PII handling, data-subject rights, retention | review |
| `reliability-reviewer` | Resilience to external-dependency failure (timeouts, retry, degradation) | review |
| `ops-reviewer` | Telemetry-spec conformance — monitor/diagnose in prod | review |
| `ui-reviewer` | Visual/UX correctness against the design pack | review |
| `a11y-reviewer` | Accessibility (WCAG, contrast, keyboard, ARIA, focus order) | review |
| `infra-reviewer` | Deployment/config correctness, secrets, exposure, resource limits | review |
| `unit-test` / `system-test` | Coverage targets; primary-flow verification | test |

**Principle:** every role is **generic + stack-aware** — there is no "stack-only" tier. `a11y-reviewer` is universal (every UX project needs it); it just needs to know *how* accessibility shows up in *this* stack. So all roles use the same layering uniformly.

## 3. Architecture — three layers (implement two first)

| Layer | Owner | Upgrade behaviour | Example (`security-reviewer`) |
|---|---|---|---|
| **Core shell** | HOS, generic | refreshed on upgrade (unless edited → squash) | "find auth bypass, injection, authz, OWASP" |
| **Stack pack** (`django`, `rails`, …) | HOS-blessed, **reusable** | refreshed independently | "Django: `select_for_update` races, tenant managers, CSRF-on-HTMX, TOTP" |
| **Project** | customer | **never touched by HOS** | "CPS's specific tenancy rule X" |

- The **stack pack is the reusable unit** — `django-pack` is extracted from CPS's reality and reused by the next Django project. This is what keeps domain knowledge from being locked to one repo.
- **Tradeoff (agreed):** model three layers conceptually; **implement core + extension first** (extension = composed stack+project). Promote the stack pack to a *separately distributable* artifact once the two-layer mechanics are proven. Avoids building a package manager on day one.

## 4. Mechanism — marker-delimited regions in one file

A Claude Code subagent is a single `.md`. Compose the layers as marker-delimited regions — the **same proven pattern** as the `CLAUDE.md` orchestrator block the installer already manages:

```markdown
---
name: security-reviewer
---
<!-- HOS:CORE:START v=<sha> — managed by HOS; overwritten on upgrade/squash. -->
[generic role definition, workflow, escalation, sign-off discipline]
<!-- HOS:CORE:END -->

<!-- HOS:PACK:django:START v=<sha> — from django-pack; refreshed with the pack. -->
[Django/HTMX/Postgres-specific checks]
<!-- HOS:PACK:django:END -->

## Project Extensions (yours — HOS never touches this)
<!-- HOS:PROJECT:START -->
[CPS-specific rules]
<!-- HOS:PROJECT:END -->
```

- Each region's content sha is recorded in `.hos-manifest` (per-region, not per-file).
- The **CORE** and any **PACK** regions are HOS/pack-managed; the **PROJECT** region is inviolable.

## 5. Install / upgrade / squash

Built on the existing `.hos-manifest` sha tracking (the prune feature already computes "unmodified since install").

- **Blank install:** write CORE + chosen PACK + an empty PROJECT stub → a complete, working team.
- **Upgrade, region unedited** (installed sha == manifest sha) → refresh that region (CORE from HOS, PACK from the pack).
- **Upgrade, region edited** → **preserve + warn + offer `--squash`** (rewrites that region, still preserving PROJECT).
- **`--squash`** (opt-in) → force-refresh CORE/PACK regions; **never** touches PROJECT. So squash is non-lossy: it only resets the generic layers a customer shouldn't have edited.
- **PROJECT region:** never read, never written by HOS.

This is exactly the original "upgrade the worker agents unless modified, then offer squash" ask — made safe by the layer boundary.

## 6. `hos-dev-pack` & dogfooding (#176)

The pack model covers **development contexts**, not just app stacks. The 4 framework-dev validators excluded from consumer installs in v0.2.2 (`framework-validator`, `doc-validator`, `spec-compliance-validator`, `framework-setup-validator`) are the seed of **`hos-dev-pack`**:

> Building a Django app = `core + django-pack`. **Building HOS itself = `core + hos-dev-pack`.**

`post-change-sweep` already has the seam (v0.2.2): its framework-review track dispatches `framework-validator` **only if installed**, else flags to a human. With `hos-dev-pack` installed, that track is fully automatic — HOS reviewing its own framework changes through its own pipeline. **That is the dogfood.**

## 7. The completeness gate (the real #225-class invariant)

v0.2.2's "manifest == install" is tautological (both read one file) — it *relocated* the dispatched-but-unshipped bug class, didn't close it. The true invariant:

> **Every agent a shipped agent dispatches must itself be shipped, or explicitly declared consumer-owned.**

Build a gate that scans shipped agents' dispatch references (`Invoke \`<agent>\``, subagent declarations) and asserts this. It would have caught the `post-change-sweep → framework-validator` break automatically. Ships as part of the base-agent work (likely in `hos-dev-pack`).

## 8. Build sequencing — bootstrap the dev tools first (agreed)

Turn v0.3.0's size into the build method:

1. **Position spec** (this doc) — reviewed before any authoring.
2. **Phase 0 — hand-bootstrap:** author the base-agent **cores + `hos-dev-pack` as plain `.claude/agents/*.md` files** (no layering machinery yet). Immediately usable — HOS starts dogfooding **by hand**.
3. **Phase 1 — dogfood:** build the rest of v0.3.0 (layering mechanics, machine accounts, completeness gate) **through** those agents. The machinery is validated by the thing it manages.
4. **Phase 2 — mechanics:** core+extension install + sha upgrade + `--squash` + completeness gate.
5. **Phase 3 — `django-pack`:** extract from CPS; CPS migrates onto `core + django-pack` (squash becomes non-lossy for CPS).

---

## 9. Relationship to machine accounts (#152)

Bundled in v0.3.0; **no hard ordering** between them. The agent pipeline can dogfood under today's identity; machine accounts add attribution/forge-proofing on top. The base-agent rework touches the *content* of agents; machine accounts touch *who authenticates* their actions. They compose.

## 10. Open questions for review

1. **Region granularity** — CORE + one PACK + PROJECT (3 regions), or allow multiple stacked packs? (Start: one pack.)
2. **Pack distribution** — packs as in-repo directories the installer composes, vs. a fetched artifact like releases? (Start: in-repo; formalize later.)
3. **Canonical set boundary** — ship all 13 roles, or a smaller "always" core (coder + the 4 universal reviewers + tests) with the rest opt-in? (Lean: ship all 13; they're all generic.)
4. **CPS migration** — assisted auto-split of CPS's flat agents into core+pack+project, or one-time manual? (Lean: assisted, but manual-reviewed.)
5. **Stack-pack ownership** — HOS-blessed only, or a community/registry path eventually? (Out of scope for v0.3.0; note the seam.)
