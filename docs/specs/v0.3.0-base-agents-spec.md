## 1. Problem & vision

HOS ships the **oversight layer** but never shipped the **base/worker team** (coder, reviewers, architect…) — those were born Django-specific in the CPS pilot and never canonicalized. So a blank install gives no working team, HOS's role expectations are implicit/scattered, and there's no upgrade path for base agents (a consumer's bug-workarounds can't be cleaned up, improvements can't be pushed).

**Vision:** HOS ships and owns a **canonical base team**. A blank repo gets a **structurally complete pipeline enforcing generic best practices**; **stack-specific depth comes from a pack** (#237 — the bare core is real but shallow; the installer warns when no pack is installed and recommends installing one before first use). Domain specifics live in a layer HOS never overwrites. The same mechanism lets HOS **dogfood its own pipeline** to build itself.

## 3. Architecture — three regions from day one (#231 resolved)

**Decision (#231):** the on-disk format is **three regions from the first install** — `CORE`, zero-or-more `PACK:<name>`, and `PROJECT`. There is no separate "two-layer" file format. "Implement incrementally" means: **start with one pack** and HOS-blessed packs only (multi-pack and external pack distribution are deferred, §10) — *not* a different file layout. The parser, sha schema, and upgrade logic are the three-region ones throughout.

| Region | Owner | Upgrade behaviour |
|---|---|---|
| `CORE` | HOS, generic | refreshed from HOS (unless edited → squash) |
| `PACK:<name>` | pack-owned (`django`, `hos-dev`, …), reusable | refreshed from the installed pack |
| `PROJECT` | customer | **never written** by HOS |

## 4. Mechanism — marker-delimited regions (#235, #236 resolved)

A subagent is a single `.md`. Compose regions with markers; **canonical order is `CORE` → `PACK:<name>` (alphabetical) → `PROJECT` last** (#235 — LLM precedence is recency-weighted, so the most-specific layer must come last so PROJECT overrides PACK overrides CORE). The installer **rewrites the file in canonical order on every upgrade.**

```markdown
---
name: security-reviewer
dispatches: [code-reviewer]        # structured dispatch declaration (#234)
---
<!-- HOS:CORE:START -->
[generic role definition, workflow, escalation, sign-off discipline]
<!-- HOS:CORE:END -->

<!-- HOS:PACK:django:START -->
[Django/HTMX/Postgres-specific checks]
<!-- HOS:PACK:django:END -->

## Project Extensions (yours — HOS never writes here)
<!-- HOS:PROJECT:START -->
[CPS-specific rules]
<!-- HOS:PROJECT:END -->
```

**Hashing & schema (#230).** `.hos-manifest` gains region rows: `path<TAB>region<TAB>sha256` (region ∈ `CORE` | `PACK:<name>` | `PROJECT` | whole-file for non-agent files, back-compatible — a 2-column row is read as whole-file). The hash is over the **raw bytes strictly between the marker lines** (marker lines themselves excluded, so a marker-format tweak doesn't churn the sha), with the trailing newline normalized to exactly one. **A flat `.md` with no markers is treated as an implicit single `CORE` region** (this is what makes Phase-0 files and pre-existing consumer agents legible to the installer).

**Marker integrity (#236).** Agent `.md` files live in a write-accessible dir, and unlike `CLAUDE.md` a running agent *could* write its own definition. Protections: (a) an instruction-level constraint — **agents must not write to their own/other agents' definition files** (added to AGENTS.md + each core); (b) the installer **validates markers before any upgrade/squash** and is **fail-closed** — malformed/duplicate/unbalanced markers → halt, emit a precise error, preserve the file untouched, require a human fix. Marker survival is asserted for **installer-managed** writes only; the runtime constraint is behavioral, not OS-enforced (a known residual, noted).

## 5. Install / upgrade / squash — three-way comparison (#229 resolved)

The `.hos-manifest` stores, per region, the **base sha = the sha HOS last wrote** (install or last refresh). Upgrade is a **three-way** decision per region (#229):

| `base_sha` (manifest) vs `disk_sha` | Action on upgrade |
|---|---|
| equal → region **unedited** since HOS wrote it | refresh from HOS/pack; update `base_sha` to the new content's sha |
| different → region **consumer-edited** | **preserve + warn + offer `--squash`** |

This is the same three-way model the prune feature already uses for whole files — the spec just makes it explicit per region. A two-way "disk == new-release" check is **wrong** (it flags every HOS improvement as an edit, and silently clobbers an edit that happens to match the new release).

- **Blank install:** write `CORE` + chosen `PACK` + an empty `PROJECT` stub; record each region's `base_sha`.
- **`--squash`** (opt-in): force-refresh `CORE`/`PACK` regions even when edited; **never** touches `PROJECT`. So squash is non-lossy — it only resets the generic layers.
- **`PROJECT`:** parsed (to locate boundaries / insert packs) but **never written** (#235 — "never written," not "never read").

### 5a. Install invocation — `--pr` by default, never falls back (human, 2026-06-15)

The install must never silently do less than it claims (the #225/#238/`--pr`-fallback class). Invocation rules:

- **`--pr` is the DEFAULT** when installing into an **existing / non-empty repo** (an upgrade) — it applies the change on a branch and opens a reviewable PR, so a consumer always reviews a framework change before it hits `main`.
- **`--no-pr`** turns it off (direct in-place install) — for consumers who want it.
- **Greenfield / empty repo → PR is skipped automatically** (direct scaffold). There's nothing to review-protect on a first install into an empty repo; requiring a PR there is friction. (Detection: empty/near-empty target, or no prior `.hos-release`.)
- **`--pr` NEVER falls back to in-place.** If it cannot create the branch/PR for any reason (branch exists, no `gh`, dirty tree, …) it **fails with a clear, actionable error** and changes nothing. `--pr` means PR. (This is what dirtied a consumer's `main` on 2026-06-15 — the fallback is banned.)
- **Cumulative by construction (#238):** an upgrade always brings the target to the *complete* current framework (complete-squash of HOS-owned files), regardless of the consumer's starting version — never an additive-only delta that stamps a version it didn't fully install.

## 7. The completeness gate — two sub-cases (#233, #234 resolved)

**The invariant:** every agent a shipped agent dispatches must itself be shipped, or be explicitly consumer-owned.

**Dispatch is read from a structured declaration, not prose (#234).** Each agent declares `dispatches: [<agent>, …]` in front-matter; the gate reads *that*, not a regex over prose. (A regex for `` Invoke `x` `` would miss the canonical `post-change-sweep → framework-validator` case, which is a prose conditional — #234.) Phase 0 authors every core with its `dispatches:` list; existing agents are retrofitted. Prose mentions are documentation; the declaration is the contract.

**Two sub-cases (#233):**
- **Sub-case A — HOS-internal consistency:** runs in `hos-dev-pack` during HOS development. Asserts every dispatched HOS-owned agent ships in `consumer_agents.txt` (or is dev-only). Catches the v0.2.2 class at the source.
- **Sub-case B — consumer-facing install-time check:** ships in the **core consumer install** (not the dev pack). At install/upgrade, it verifies that every agent the installed set dispatches is present in the consumer's repo (HOS-shipped *or* a consumer/base-team agent the manifest knows about) — and warns (not hard-fail) on a gap, so a consumer upgrading into a missing-agent state gets a pre-flight signal instead of a runtime break. This is the half v1 elided.

## 8. Build sequencing — bootstrap dev tools first (#232 resolved)

1. **Position spec** (this doc) — approved before authoring.
2. **Phase 0 — hand-bootstrap:** author base-agent **cores + `hos-dev-pack` as plain (flat) `.md` files** + their `dispatches:` front-matter. Flat files are legible as implicit single-`CORE` regions (§4), so they're immediately usable — HOS dogfoods **by hand**.
3. **Phase 1 — dogfood:** build the rest of v0.3.0 (layering installer, machine accounts, completeness gate) **through** those agents.
4. **Phase 2 — mechanics:** the layering installer. **On first run it migrates flat Phase-0 files (#232): wraps their entire content in `HOS:CORE:START/END`, records the sha as `base_sha` — a one-time, content-preserving migration.** Then sha upgrade + `--squash` + both completeness sub-cases.
5. **Phase 3 — `django-pack` + CPS migration (Q4 "borg"):** **absorb everything reusable from CPS's agents into `core` + `django-pack`** so that **`core + django-pack` is a complete, working install on its own** — what we ship to CPS must work standalone. Extract aggressively: anything Django-reusable → `django-pack`; anything universal → `core`. Only genuinely CPS-unique content remains, installed by a **separate project-install step run *after* the core install** (it populates the PROJECT region; the core install never depends on it). CPS comes back via a **clean v0.3.0 install** (not an incremental upgrade) — consistent with the complete-squash model (§5) and the #238 cumulative-install fix.

## 10. Resolved decisions (human, 2026-06-15)

1. **Multiple stacked packs → allowed, but UNTESTED.** The format supports `PACK:<a>` + `PACK:<b>`; we won't exercise it for a while. The installer must not break on it, but the docs must **call it out as untested** (no support guarantee for multi-pack conflict resolution/ordering until we use it). Single pack is the proven path.
2. **Pack distribution → single in-repo distribution for now.** One distribution mechanism — packs are in-repo directories the installer composes — for pilot simplicity. **Refactor to a fetched/distributable artifact later.**
3. **Canonical set → ship all 13 roles.** Keep it simple; no always-core/opt-in split.
4. **Migration → "borg" everything reusable into core + pack; project specifics install separately, after the core.** This is the load-bearing decision: **what we ship to CPS must be complete and work.** So core + the (django) pack must absorb *everything reusable* from CPS's agents — the pack is rich, not thin. Only genuinely project-unique content is left over, and it is installed by a **separate project install step run *after* the core install** (the PROJECT region is populated by that second step, not by the core install). Consequence for the model: the **core+pack install is the complete, working unit** (it stands alone); the project install is an optional overlay. See §3, §5, §8.
5. **External / community pack registry → later.** Out of scope for v0.3.0; note the seam, don't build it.
