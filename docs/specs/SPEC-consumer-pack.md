# Requirements Spec — Consumer Pack Scaffolding and Brownfield Migration

**Document type:** Requirements specification
**Status:** Approved for build
**Issues:** #275
**Date:** 2026-06-16
**Author:** pm-agent

**Cross-reference:** `SPEC-installer-upgrade.md` §2 (post-install path cleanup) and §3 (pack placeholder substitution) apply unchanged to brownfield installs. All acceptance criteria in those sections hold for the `--brownfield` path.

---

## Problem Statement

CondoParkShare (CPS) is a v0.1.0 consumer — pre-region-model, flat agent files, no `.hos-manifest`. The standard v0.3.x installer hard-stops on these because it cannot three-way merge without a recorded base. A squash adoption (`--squash`) re-wraps the flat content as CORE and overwrites it with HOS's CORE, which is lossless only if the consumer has pre-marked their PROJECT customizations into markers first. Field report #275 shows this is non-trivial: ~14 agents carry CPS-specific depth that is not covered by HOS CORE or the django pack, and the pre-marking step is manual, error-prone, and undiscoverable.

Three related gaps follow from this:

1. No automated path to turn a pre-region-model (flat) consumer repo into a correctly-layered install without manual pre-marking.
2. No tooling to distinguish stock HOS CORE content (safe to overwrite) from genuine consumer customizations (must be preserved) in an existing flat file.
3. No scaffolding path for consumers who have deep cross-cutting customizations that logically belong in a versioned consumer pack, not in per-agent PROJECT regions.

Additionally, the CPS field test surfaced a structural gap in how dep-mapper depth is layered in pack contexts: the django pack provides generic blast-radius depth, but consumer-specific tracing rules (e.g., CPS's ORM signal dispatchers) have no designated home until a consumer pack exists.

## §1 — Brownfield Migration Flag

### Background

A "brownfield" consumer is one that has existing agent files with no `.hos-manifest` and no region markers — the v0.1.0 flat-file model. Today the installer hard-stops on these with `CORE drifted (base_sha=(absent))`. The `--brownfield` flag defines an automated migration path that replaces the manual pre-marking step.

### Requirements

**REQ-B-01 — `--brownfield` flag.**
`hos_install.sh` must accept a `--brownfield` flag. When present, the installer does not hard-stop on a missing `.hos-manifest` or on flat (marker-less) agent files. Instead it runs the duplicate-logic check (§2) against each existing flat agent file before performing the three-way merge.

`--brownfield` is mutually exclusive with `--squash`. If both are passed, the installer must exit non-zero with:
```
ERROR: --brownfield and --squash are mutually exclusive.
  --brownfield performs its own safe classification before merging.
  --squash overwrites all CORE/PACK regions without classification.
  Use --brownfield for pre-region-model repos.
```

**REQ-B-02 — No hard-stop on absent `.hos-manifest`.**
When `--brownfield` is set, an absent `.hos-manifest` is not an error. The installer treats this as a brownfield first install and constructs a synthetic baseline manifest from the duplicate-logic check output (REQ-B-05). The synthetic manifest is written to `.hos-manifest` before the three-way merge runs, so the merge has a valid baseline.

**REQ-B-03 — No hard-stop on flat agent files.**
When `--brownfield` is set, flat (marker-less) agent files that would normally produce a `CORE drifted (base_sha=(absent))` hard-stop are instead processed by the duplicate-logic check (§2). The hard-stop is suppressed for these files; the duplicate-logic check output drives the region construction.

**REQ-B-04 — Flat-file detection.**
The installer must identify which agent files in the consumer's `.claude/agents/` directory are flat (no `<!-- HOS:` markers) versus already-marked. Already-marked files are processed by the standard three-way merge. Only flat files are sent to the duplicate-logic check.

**REQ-B-05 — Synthetic baseline manifest.**
After running the duplicate-logic check against all flat files, the installer must construct a synthetic `.hos-manifest` baseline. For each flat agent file:
- Sections classified as "stock CORE" (§2 REQ-D-03) are recorded in the manifest with base-shas derived from the current HOS CORE template for that agent. This ensures the three-way merge treats them as CORE drift (resolved by the standard merge taking HOS's version).
- Sections classified as "PROJECT customization" (§2 REQ-D-04) are wrapped in PROJECT region markers and written to the disk file before the merge runs. The merge then carries them through untouched.

The synthetic manifest must be written to `.hos-manifest` and annotated with a single comment line at the top:
```
# Synthetic baseline generated by --brownfield on <ISO-8601-date>. Manual review recommended.
```

**REQ-B-06 — Normal merge after baseline.**
After the synthetic baseline is written, the installer proceeds with the standard three-way region merge (as documented in the existing installer). The `--brownfield` flag only affects the pre-merge baseline construction. The merge itself is unchanged.

**REQ-B-07 — Install log — brownfield summary.**
After completing a brownfield install, the installer must emit a structured summary section in the install log:
```
[brownfield] Migration complete.
  Agents processed:          <N>
  Already-marked (skipped):  <N>
  Flat files classified:     <N>
  PROJECT regions preserved: <N>
  Stock CORE overwritten:    <N>
  Classification report:     .claudetmp/brownfield-<YYYYMMDD-HHmmss>-report.txt
```

**REQ-B-08 — `--dry-run` compatibility.**
`--brownfield --dry-run` must show what the duplicate-logic check would classify and what the synthetic baseline would contain, without writing any files. The classification report is written to `.claudetmp/` even in dry-run (it is read-only output, not a state change).

**Acceptance criteria for §1:**

| ID | Criterion |
|---|---|
| AC-B-01 | A v0.1.0 consumer repo (flat agent files, no `.hos-manifest`) completes `hos_install.sh --local --brownfield --pack django` without a hard-stop, and every agent file in `.claude/agents/` has correct CORE / PACK:django / PROJECT regions. |
| AC-B-02 | Content in the consumer's flat files that the duplicate-logic check classifies as "PROJECT customization" is present in the corresponding agent's PROJECT region after the install. |
| AC-B-03 | Content that the duplicate-logic check classifies as "stock CORE" is NOT present in the PROJECT region; the installed CORE region matches HOS's shipped version. |
| AC-B-04 | The install log emits the `[brownfield] Migration complete.` summary block with correct counts. |
| AC-B-05 | A `.hos-manifest` is written after the install and is valid (parseable by `regions.py`). |
| AC-B-06 | `--brownfield --squash` exits non-zero with the mutual-exclusion error message. |
| AC-B-07 | `--brownfield --dry-run` produces the classification report in `.claudetmp/` without modifying any agent files or writing `.hos-manifest`. |
| AC-B-08 | Agent files already carrying region markers (not flat) are processed by the standard three-way merge, not the duplicate-logic check. |

## §2 — Duplicate-Logic Check

### Background

The duplicate-logic check is the mechanism that differentiates stock HOS CORE content from genuine consumer customizations in a flat agent file. It avoids requiring the consumer to manually pre-mark their PROJECT regions before a brownfield install.

### Algorithm

The comparison is line-diff ratio. No NLP or semantic analysis is required for v1.

**REQ-D-01 — Unit of comparison: section.**
A "section" is a contiguous block of lines that begins at a heading (a line starting with `#`) or at the start of the file, and ends at the next heading or end of file. The duplicate-logic check compares sections from the consumer's flat file against corresponding sections from the current HOS CORE template for that agent.

Matching is by heading text (case-insensitive, stripped of leading `#` and whitespace). A section in the consumer file that has no corresponding section in the CORE template is classified as "PROJECT customization" by default.

**REQ-D-02 — Similarity metric.**
For each matched section pair (consumer section vs. CORE template section), compute the line-diff ratio:

```
similarity = (lines in common) / (max(len(consumer_section), len(core_template_section)))
```

"Lines in common" is the count of lines that appear in both the consumer section and the CORE template section, counting each line once (set intersection by content, stripping leading and trailing whitespace before comparison).

This is a simple set-overlap ratio, not a sequence alignment. It is deliberately unsophisticated for v1; no semantic similarity or embedding-based comparison is performed.

**REQ-D-03 — "Stock CORE" threshold.**
A section is classified as "stock CORE" if its similarity score is >= 0.90 (90%). The installer treats stock CORE sections as safe to overwrite with the HOS CORE template.

**REQ-D-04 — "PROJECT customization" threshold.**
A section is classified as "PROJECT customization" if its similarity score is < 0.90. The installer preserves PROJECT customization sections by placing them in the agent's PROJECT region in the synthetic baseline.

**REQ-D-05 — Edge case: 50/50 split sections.**
A section whose similarity score falls in the range [0.45, 0.90) — meaning it is neither clearly stock nor clearly custom — is classified as "PROJECT customization" (the conservative choice: never silently drop consumer content). The classification report marks these sections with a `[mixed]` tag so the human can review them. The installer does not block or warn at the gate on mixed sections; it classifies conservatively and surfaces them in the report.

A section with similarity score < 0.45 is classified as "PROJECT customization" without the `[mixed]` tag (it is clearly custom).

**REQ-D-06 — Classification report format.**
The duplicate-logic check must produce a classification report for each processed flat file. The report is written to `.claudetmp/brownfield-<YYYYMMDD-HHmmss>-report.txt` and is also included in the install log as a human-readable summary.

Per-file report format:
```
Agent: <agent-filename>
  Section: "<heading text>"
    CORE template match: <yes|no>
    Similarity: <0.00–1.00>
    Classification: STOCK_CORE | PROJECT_CUSTOMIZATION | PROJECT_CUSTOMIZATION [mixed]
  ...
  Summary: <N> sections → <M> stock, <P> customization (<Q> mixed)
```

**REQ-D-07 — Machine-readable classification output.**
In addition to the human-readable report, the duplicate-logic check must produce a machine-readable JSON summary per agent, consumed by the installer to construct the synthetic baseline (REQ-B-05). Format:

```json
{
  "agent": "<agent-filename>",
  "sections": [
    {
      "heading": "<heading text>",
      "classification": "STOCK_CORE | PROJECT_CUSTOMIZATION",
      "mixed": true | false,
      "similarity": 0.00
    }
  ]
}
```

**REQ-D-08 — No CORE template → classify as PROJECT.**
If the HOS source does not ship a CORE template for a given agent slug (the agent is consumer-only, not in `consumer_agents.txt`), the entire flat file is classified as "PROJECT customization". The check does not attempt comparison against a missing template.

**REQ-D-09 — Comparison scope.**
The duplicate-logic check compares the body text of sections (not front-matter: not the `description:`, `tools:`, or `model:` YAML front-matter lines). Front-matter is always taken from the HOS CORE template on install, consistent with the current three-way merge behavior. The duplicate-logic check does not attempt to classify or preserve front-matter content from the flat file.

**Acceptance criteria for §2:**

| ID | Criterion |
|---|---|
| AC-D-01 | A flat agent file whose content is byte-identical to the HOS CORE template for that agent produces a classification report showing 100% similarity for every section, all classified `STOCK_CORE`. |
| AC-D-02 | A flat agent file with one section that has no counterpart in the CORE template classifies that section as `PROJECT_CUSTOMIZATION`. |
| AC-D-03 | A flat agent file with a section that is 85% similar to the CORE template (below the 90% threshold) classifies that section as `PROJECT_CUSTOMIZATION [mixed]`. |
| AC-D-04 | A flat agent file with a section that is 40% similar to the CORE template classifies that section as `PROJECT_CUSTOMIZATION` (no `[mixed]` tag). |
| AC-D-05 | After a brownfield install of such an agent, the `[mixed]`-tagged section appears in the agent's PROJECT region, and the install log notes the mixed count. |
| AC-D-06 | The classification report is written to `.claudetmp/` and is valid JSON for the machine-readable portion. |
| AC-D-07 | An agent slug not in `consumer_agents.txt` produces a report classifying the entire file as `PROJECT_CUSTOMIZATION` with a `no CORE template` note. |
| AC-D-08 | Front-matter lines (`description:`, `tools:`, `model:`) do not appear in the classification report — they are always sourced from HOS. |

## §3 — Consumer Pack Scaffolding

### Background

A consumer who has deep, cross-cutting customizations across many agents for the same stack is better served by a versioned consumer pack (e.g., `PACK:condoparkshare`) than by per-agent PROJECT regions. A consumer pack survives upgrades as a unit, can be versioned independently, and is composable on top of the base django pack. This scaffolding feature is opt-in, offered after the brownfield duplicate-logic check identifies significant customization volume.

### Trigger Condition

**REQ-CS-01 — Offer scaffold when threshold is met.**
After the duplicate-logic check completes, if three or more agent files each have one or more sections classified as "PROJECT customization", the installer must offer to scaffold a consumer pack. The offer is presented as an interactive prompt when running interactively; in non-interactive / CI mode (`--non-interactive` or when stdin is not a tty) it is skipped and the migration proceeds with PROJECT regions only.

Prompt text:
```
[brownfield] Found PROJECT customizations in <N> agents.
  These could be extracted into a consumer pack for cleaner versioning.
  Scaffold a consumer pack? (recommended for N >= 3) [y/N]:
```

The default is `N` (no scaffolding). The consumer must opt in explicitly. The `--scaffold-pack <slug>` flag on `hos_install.sh` pre-answers yes and provides the slug, enabling non-interactive scaffolding in scripts.

**REQ-CS-02 — Consumer pack directory structure.**
When scaffolding is accepted, the installer creates the following structure in the consumer's target repo:

```
packs/<slug>/
  pack.toml
  <agent>.md   (one file per agent with classified PROJECT content)
```

Where `<slug>` is either the value passed to `--scaffold-pack` or the value entered at the interactive prompt. The slug must match the pattern `[a-z][a-z0-9-]*` (lowercase, alphanumeric, hyphens). The installer must reject slugs that conflict with HOS built-in pack names (e.g., `django`, `testpack`).

**REQ-CS-03 — Minimum viable `pack.toml`.**
The scaffolded `pack.toml` must contain the following fields, all required:

```toml
name = "<slug>"
description = "<Consumer project name> — project-specific HOS pack."
version = "0.1.0"
requires = []
```

The `requires` field is an array of pack names that this pack depends on. For a consumer who is also using the django pack, the human should add `"django"` to `requires` after scaffolding; the installer does not auto-populate it. A comment in the scaffolded `pack.toml` instructs the human:
```toml
# If this pack assumes another pack (e.g. django), add it to requires:
# requires = ["django"]
```

The `requires` field is informational in v0.4.0. The installer does not enforce composition ordering based on `requires`; it is a documentation signal for humans and for future tooling.

**REQ-CS-04 — Agent body files.**
For each agent with classified PROJECT customization sections, the installer creates `packs/<slug>/<agent>.md` containing only the PROJECT customization content — the text of each classified section, joined in the order they appeared in the original flat file.

The body file must NOT include:
- Region markers (`<!-- HOS:CORE:START -->` etc.) — body files are marker-free by convention (they are injected into the appropriate marker by `regions.py inject-pack`).
- Front-matter (the YAML `description:`/`tools:`/`model:` block).
- Stock CORE content.

If an agent's classified PROJECT content is empty after excluding the above (all sections were stock CORE), no body file is created for that agent and no entry appears in `packs/<slug>/`.

**REQ-CS-05 — PROJECT regions become pack after scaffolding.**
When the installer scaffolds a consumer pack, the PROJECT regions it would otherwise have written to the individual agent files are instead registered as the pack's body files (REQ-CS-04). The agent files receive empty PROJECT region stubs (`<!-- HOS:PROJECT:START -->` … `<!-- HOS:PROJECT:END -->`), not the classified content. The consumer then installs the scaffolded pack by re-running:
```
hos_install.sh --pack <slug> --pack django [DIR]
```

A post-scaffolding message in the install log instructs the consumer on this step:
```
[brownfield] Consumer pack scaffolded at packs/<slug>/.
  To apply it, re-run the installer with --pack <slug> (and any other packs).
  Example: hos_install.sh --pack django --pack <slug> [DIR]
```

**REQ-CS-06 — Consumer pack resolution path.**
The installer must resolve packs from two locations, in priority order:
1. The consumer repo's own `packs/<name>/` directory (consumer-local packs).
2. The HOS source's `packs/<name>/` directory (HOS-shipped packs).

If a pack name exists in both locations, the consumer-local version takes precedence. This is the mechanism by which a scaffolded consumer pack is usable immediately without being published to the HOS repo.

The install log must emit the resolution source for each pack:
```
[pack] Resolved <name> from consumer-local packs/ (not HOS-shipped)
```
or
```
[pack] Resolved <name> from HOS source (HOS-shipped)
```

**REQ-CS-07 — `--scaffold-pack` is not valid without `--brownfield`.**
If `--scaffold-pack` is passed without `--brownfield`, the installer must exit non-zero:
```
ERROR: --scaffold-pack requires --brownfield.
  Consumer pack scaffolding is part of the brownfield migration flow.
```

**REQ-CS-08 — Scaffolded pack is not auto-committed.**
The installer writes the `packs/<slug>/` directory to the consumer repo but does not add, stage, or commit it. The human is responsible for reviewing and committing. The post-scaffolding message (REQ-CS-05) includes a reminder.

**Acceptance criteria for §3:**

| ID | Criterion |
|---|---|
| AC-CS-01 | A brownfield install where the duplicate-logic check finds PROJECT customizations in 5 agents offers the scaffold prompt. |
| AC-CS-02 | A brownfield install where the check finds PROJECT customizations in 2 agents does not offer the scaffold prompt. |
| AC-CS-03 | After accepting the scaffold with slug `condoparkshare`, `packs/condoparkshare/pack.toml` exists with `name = "condoparkshare"` and `version = "0.1.0"`. |
| AC-CS-04 | `packs/condoparkshare/<agent>.md` exists for each agent with classified PROJECT content, and its content does not include region markers or front-matter. |
| AC-CS-05 | The corresponding agent files have empty PROJECT region stubs (not the classified content) after scaffolding. |
| AC-CS-06 | Re-running `hos_install.sh --pack django --pack condoparkshare` after scaffolding correctly injects the consumer pack body as `PACK:condoparkshare` regions. |
| AC-CS-07 | The install log shows `[pack] Resolved condoparkshare from consumer-local packs/`. |
| AC-CS-08 | Passing `--scaffold-pack condoparkshare` without `--brownfield` exits non-zero with the required error message. |
| AC-CS-09 | A consumer-local pack name that conflicts with a HOS built-in pack name (e.g., `django`) is rejected with an error message. |
| AC-CS-10 | The scaffolded files are not staged or committed by the installer. |

## §4 — dep-mapper Depth Convention

### Background

CPS field report #275 (Q2) asked where CPS-specific dep-mapper tracing rules belong — in a PROJECT region or in a pack — given that dep-mapper is handled specially by the installer (`--force` replaces it wholesale, and the standard path preserves it if the consumer has a project-specific version). The django pack does not ship a `dep-mapper.md` body file. This gap means a consumer adopting the django pack gets no dep-mapper depth from the pack and must carry all tracing logic in dep-mapper's PROJECT region.

The convention below formalizes where each level of dep-mapper depth lives.

### Convention

**REQ-DM-01 — Base pack provides generic framework depth.**
A base pack (e.g., `django`) may ship `packs/django/dep-mapper.md` to provide generic Django ORM blast-radius tracing depth — for example, identifying related ORM lookups, signal dispatcher chains, and cache invalidation patterns that apply to any Django project. This is the stack-aware but project-agnostic layer.

The django pack does not currently ship a `dep-mapper.md`. Adding one is an additive change to the django pack; this spec defines the convention so the django pack maintainer can fill it without a separate spec.

**REQ-DM-02 — Consumer pack provides project-specific tracing.**
A consumer pack (e.g., `packs/condoparkshare/`) may ship `packs/condoparkshare/dep-mapper.md` to provide project-specific tracing rules: custom model relationships, non-standard signal patterns, project-specific cache keys, or any tracing depth that is meaningful only for that consumer's codebase. This is the stack-aware and project-specific layer.

When both a base pack and a consumer pack ship a `dep-mapper.md`, the installer injects both as separate PACK regions: `PACK:django` and `PACK:condoparkshare`. The consumer pack's region is injected after the base pack's region, and its content extends or overrides the base pack's depth for the specific patterns it names. There is no automated conflict resolution between pack regions; the consumer pack author is responsible for ensuring the two layers are coherent.

**REQ-DM-03 — PROJECT region for one-off tracing.**
Until a consumer pack is scaffolded, the dep-mapper agent's PROJECT region is the designated location for consumer-specific tracing rules. This is the supported interim path documented in the issue #275 guidance.

When a consumer pack is subsequently scaffolded via `--scaffold-pack`, the classified PROJECT content from dep-mapper is extracted into `packs/<slug>/dep-mapper.md` following the same body-file conventions as other agents (REQ-CS-04).

**REQ-DM-04 — `--force` behavior is unchanged.**
The existing `--force` behavior for dep-mapper (overwrite with the HOS-shipped generic version) is unchanged by this spec. Consumers who want to preserve their PROJECT region or pack depth must use the three-way merge path (without `--force`), or pre-mark their content into a PROJECT region or consumer pack before running with `--force`.

The `--brownfield` path does not pass `--force` to dep-mapper by default. In a brownfield install, the consumer's existing dep-mapper file is processed by the duplicate-logic check like any other agent.

**REQ-DM-05 — Consumer-pack install documentation.**
`docs/UPGRADE-PR-REVIEW-CHECKLIST.md` must be updated to include a section on consumer packs covering:
- How to resolve dep-mapper depth (PROJECT region until a consumer pack is scaffolded; consumer pack body file after).
- How to verify that the consumer pack was resolved from `consumer-local packs/` and not from HOS source.
- How to confirm that both a base pack and a consumer pack inject correctly into dep-mapper when both ship `dep-mapper.md`.

**Acceptance criteria for §4:**

| ID | Criterion |
|---|---|
| AC-DM-01 | When both `packs/django/dep-mapper.md` and `packs/condoparkshare/dep-mapper.md` exist, the installed dep-mapper agent contains both `<!-- HOS:PACK:django:START -->` and `<!-- HOS:PACK:condoparkshare:START -->` regions. |
| AC-DM-02 | In a brownfield install (without `--force`), the consumer's existing dep-mapper file is processed by the duplicate-logic check, not replaced wholesale. |
| AC-DM-03 | After brownfield install with pack scaffolding accepted, dep-mapper's classified PROJECT content appears in `packs/<slug>/dep-mapper.md` and dep-mapper's PROJECT region in the agent file is the empty stub. |
| AC-DM-04 | `docs/UPGRADE-PR-REVIEW-CHECKLIST.md` includes a "Consumer packs" section covering the three points in REQ-DM-05. |

## Out of Scope

- **Empty PROJECT region stubs (#303 Finding 6):** Deferred per `SPEC-installer-upgrade.md`. The empty stub written by `--first-install` already exists; stub-on-brownfield is covered by REQ-B-05 and REQ-CS-05 above.
- **`requires` enforcement and multi-pack composition ordering:** Informational in v0.4.0. The `requires` field in `pack.toml` is documented but not mechanically enforced by the installer. This is a v0.5.0 item.
- **Publishing a consumer pack to the HOS registry:** Out of scope. Consumer packs are consumer-local in v0.4.0; no publish or distribution mechanism is specified here.
- **Semantic / NLP-based similarity in the duplicate-logic check:** Out of scope for v1. Line-diff ratio (§2 REQ-D-02) is the algorithm for this spec. A more sophisticated classifier may be introduced in a later spec.
- **Front-matter preservation from flat files:** Out of scope. Front-matter is always sourced from the HOS CORE template. Any consumer front-matter customizations are surfaced in the classification report for human review but are not preserved automatically.

## Implementation Notes for Architect

These are observations to inform architecture and technical-design decisions. They are not requirements.

- The duplicate-logic check algorithm (REQ-D-02) uses set-intersection similarity per section. Heading-based section splitting may fail on agent files that have no headings (pure prose). Technical design should define a fallback: treat a heading-less file as a single section and compare it against the entire CORE template body.
- Consumer-local pack resolution (REQ-CS-06) requires the installer to search two pack directories per pack name. The installer already resolves pack bodies with `$HOS_SOURCE/packs/$_pk/${agent}.md`. Technical design must define the consumer-local resolution path (`$TARGET_REPO/packs/$_pk/${agent}.md`) and the precedence logic (consumer-local first, then HOS source).
- The `--scaffold-pack` flag implies a re-run of the installer after scaffolding to inject the pack. Technical design should evaluate whether a single-run path is feasible (scaffold + inject in one pass) without complicating the region merge ordering.
- The `requires` field in `pack.toml` is not enforced, but the installer should parse it and warn if a declared dependency is not also passed as `--pack`. This is a quality-of-life check, not a hard-stop, and is a technical-design decision.
